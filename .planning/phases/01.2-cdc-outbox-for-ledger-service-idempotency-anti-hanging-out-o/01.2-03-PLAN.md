**HungTransferDetector.java** (per D-17, D-18, D-19, D-21): Create at `payment-service/src/main/java/au/com/bankforge/payment/detector/HungTransferDetector.java`:

```java
package au.com.bankforge.payment.detector;

import au.com.bankforge.common.enums.TransferState;
import au.com.bankforge.payment.entity.Transfer;
import au.com.bankforge.payment.repository.TransferRepository;
import au.com.bankforge.payment.service.TransferStateService;
import lombok.RequiredArgsConstructor;
import lombok.extern.slf4j.Slf4j;
import org.springframework.kafka.core.KafkaTemplate;
import org.springframework.scheduling.annotation.Scheduled;
import org.springframework.stereotype.Component;

import java.time.Duration;
import java.time.Instant;
import java.time.temporal.ChronoUnit;
import java.util.List;

@Component
@RequiredArgsConstructor
@Slf4j
public class HungTransferDetector {

    private final TransferRepository transferRepository;
    private final TransferStateService transferStateService;
    private final KafkaTemplate<String, String> kafkaTemplate;

    @Scheduled(fixedDelay = 60_000)
    public void detectHungTransfers() {
        detectHungPaymentProcessing();
        detectHungPosting();
    }

    void detectHungPaymentProcessing() {
        Instant threshold = Instant.now().minus(5, ChronoUnit.MINUTES);
        List<Transfer> hung = transferRepository
            .findByStateAndCreatedAtBefore(TransferState.PAYMENT_PROCESSING, threshold);
        for (Transfer t : hung) {
            log.warn("Hung PAYMENT_PROCESSING transfer: transferId={} age={}",
                t.getId(), Duration.between(t.getCreatedAt(), Instant.now()));
            transferStateService.cancel(t.getId(), "PAYMENT_PROCESSING_TIMEOUT");
            String failPayload = "{\"transferId\":\"" + t.getId()
                + "\",\"reason\":\"PAYMENT_PROCESSING_TIMEOUT\"}";
            kafkaTemplate.send("banking.transfer.failed", t.getId().toString(), failPayload);
            log.info("Published banking.transfer.failed for transferId={}", t.getId());
        }
    }

    void detectHungPosting() {
        Instant threshold = Instant.now().minus(15, ChronoUnit.MINUTES);
        List<Transfer> hung = transferRepository
            .findByStateAndUpdatedAtBefore(TransferState.POSTING, threshold);
        for (Transfer t : hung) {
            log.error("Hung POSTING transfer (manual review required): transferId={} age={}",
                t.getId(), Duration.between(t.getUpdatedAt(), Instant.now()));
            // Direct state set -- NOT via FSM transition.
            // FAILED is an operational exceptional state used only by the scheduler.
            // FSM bypass is intentional: avoids polluting the normal transfer lifecycle
            // with a transition that only this edge-case path uses.
            t.setState(TransferState.FAILED);
            t.setErrorMessage("LEDGER_CONFIRM_TIMEOUT -- manual review required");
            transferRepository.save(t);
            // D-19: Do NOT publish reversal event or trigger compensation.
            // Ledger entries are already written (balanced). Only the confirmation
            // event is missing. Manual review is the correct action.
        }
    }
}
```

Key behavioral rules:
- PAYMENT_PROCESSING: `createdAt < now() - 5 minutes` -> cancel via `TransferStateService.cancel()` -> publish `banking.transfer.failed`
- POSTING: `updatedAt < now() - 15 minutes` -> direct `setState(FAILED)` + save -> NO reversal event, NO compensation
- The `banking.transfer.failed` topic is auto-created by Spring Kafka (D-21)

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01.2-08 | Tampering | HungTransferDetector POSTING->FAILED | accept | Direct setState bypasses FSM but is intentional (Research Pattern 5). Transfer entity has no @PreUpdate validation of state transitions -- only @PreUpdate sets updatedAt. No external input can trigger this path; only the scheduler can. |
| T-01.2-09 | Denial of Service | @Scheduled(fixedDelay = 60_000) | accept | Fixed delay (not fixed rate) ensures the next run waits 60s AFTER the previous run completes. Long-running queries do not cause overlapping executions. |
| T-01.2-10 | Repudiation | POSTING timeout not publishing reversal | accept | By design (D-19). The POSTING timeout is logged at ERROR level with transferId and duration. The `LEDGER_CONFIRM_TIMEOUT -- manual review required` error message is stored on the Transfer entity in the database. This is auditable. |
</threat_model>
