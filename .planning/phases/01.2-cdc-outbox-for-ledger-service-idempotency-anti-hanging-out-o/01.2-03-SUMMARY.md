# 01.2-03 Summary: Hung-Transfer Detector

## Objective
Implement a `@Scheduled` job in payment-service that detects stuck PAYMENT_PROCESSING and POSTING transfers and resolves them according to per-state rules (D-19).

## Key Files

### Created
- `payment-service/src/main/java/au/com/bankforge/payment/detector/HungTransferDetector.java` — `@Scheduled(fixedDelay=60s)` job; PAYMENT_PROCESSING >5 min → `TransferStateService.cancel()` + publish `banking.transfer.failed`; POSTING >15 min → direct `setState(FAILED)` + save, no reversal (D-19)
- `payment-service/src/test/java/au/com/bankforge/payment/detector/HungTransferDetectorTest.java` — 3 unit tests: cancel hung PAYMENT_PROCESSING, mark hung POSTING as FAILED, no-op when no hung transfers

### Modified
- `payment-service/src/main/java/au/com/bankforge/payment/repository/TransferRepository.java` — added `findByStateAndCreatedAtBefore` and `findByStateAndUpdatedAtBefore` Spring Data JPA methods
- `payment-service/src/main/java/au/com/bankforge/payment/PaymentServiceApplication.java` — added `@EnableScheduling`

## Decisions & Deviations

- **D-19 FSM bypass for POSTING:** Direct `setState(FAILED)` bypasses the state machine intentionally. The ledger has already posted; running FSM compensation would double-reverse the ledger entry. No reversal event published.
- **PAYMENT_PROCESSING uses `createdAt`:** The transfer hasn't reached the ledger yet, so `createdAt` is the correct timeout anchor (not `updatedAt`).
- **POSTING uses `updatedAt`:** The transfer is awaiting ledger confirmation; `updatedAt` reflects the last state transition into POSTING.
- **Configurable delay:** `fixedDelayString` backed by `payment.hung-transfer-detector.fixed-delay-ms` property (defaults to 60 000 ms) for testability.
