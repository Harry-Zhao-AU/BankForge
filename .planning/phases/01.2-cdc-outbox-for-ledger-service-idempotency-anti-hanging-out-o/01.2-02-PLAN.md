```java
package au.com.bankforge.ledger.kafka;

import lombok.extern.slf4j.Slf4j;
import org.apache.kafka.clients.consumer.ConsumerRecord;
import org.springframework.kafka.annotation.KafkaListener;
import org.springframework.stereotype.Component;

@Component
@Slf4j
public class DltHandler {

    @KafkaListener(
        topics = "banking.transfer.events.DLT",
        groupId = "ledger-service-dlt"
    )
    public void onDltMessage(ConsumerRecord<String, String> record) {
        log.error("DLT message received: topic={} partition={} offset={} key={} value={}",
            record.topic(),
            record.partition(),
            record.offset(),
            record.key(),
            record.value());

        // Extract original topic/partition/offset from headers if present
        record.headers().forEach(header ->
            log.error("  DLT header: {}={}", header.key(), new String(header.value()))
        );
    }
}
```

Key points per D-12, D-13:
- `groupId = "ledger-service-dlt"` -- separate consumer group from the main listener
- Logs structured ERROR with all record metadata
- Iterates Spring Kafka DLT headers (`kafka_original-topic`, `kafka_original-partition`, `kafka_original-offset`, `kafka_exception-fqcn`, `kafka_exception-message`) -- these are added automatically by `DeadLetterPublishingRecoverer` (D-12)
- This handler does NOT re-publish consumed messages (prevents feedback loop -- security mitigation)

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Kafka consumer -> LedgerEventListener | Untrusted payload from Kafka (could be malformed or poison pill) |
| DLT topic -> DltHandler | Exhausted-retry messages; handler must not re-publish (feedback loop risk) |
| client -> PaymentController | X-Transaction-Id header exposes internal transferId (UUID, non-sensitive) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01.2-04 | Denial of Service | DltHandler | mitigate | DltHandler is read-only: logs headers and value, does NOT call `kafkaTemplate.send()` or re-publish to any topic. Prevents DLT feedback loop where consumed DLT messages trigger new errors that land back in DLT. |
| T-01.2-05 | Spoofing | LedgerEventListener transferId parsing | mitigate | `UUID.fromString(transferIdNode.asText())` already wrapped in try/catch via the existing `parsePayload` + null checks. Malformed transferId throws `IllegalArgumentException` which triggers retry -> DLT routing. |
| T-01.2-06 | Information Disclosure | X-Transaction-Id header | accept | Exposes UUID transferId in HTTP response header. UUIDs are non-sequential (v4) and not security-sensitive. Required by D-15 for operational observability. |
| T-01.2-07 | Denial of Service | ExponentialBackOff retry | mitigate | `maxElapsedTime=30000ms` (D-11) caps retry duration. After ~3 attempts, message routes to DLT. Prevents infinite retry loop from poison pill payloads. |
</threat_model>
