## DLT strategy for ledger-service

| Option | Description | Selected |
|--------|-------------|----------|
| @RetryableTopic | Spring Kafka declarative retry + DLT | | 
| DefaultErrorHandler + DeadLetterPublishingRecoverer | EOS-compatible DLT; manual config | ✓ |
| No DLT (accept message loss) | Current state — violates D-06 | |

**User's choice:** "just pick what the best practise in bank" — Claude selected `DeadLetterPublishingRecoverer` as the EOS-compatible approach (note: after removing EOS from ledger-service, `@RetryableTopic` would also be valid, but the recoverer pattern was already in the design and is more explicit for financial systems).

**Notes:** `@RetryableTopic` was previously excluded because it conflicts with `KafkaTransactionManager`. Since EOS is being removed from ledger-service (D-06), `@RetryableTopic` would technically be usable post-refactor, but `DeadLetterPublishingRecoverer` is preferred for banking: it gives explicit control over the DLT publish, headers, and error logging.

## Hung-transfer detection

| Option | Description | Selected |
|--------|-------------|----------|
| PAYMENT_PROCESSING only | Simpler; POSTING hangs treated out-of-band | |
| Both states, single timeout | Same timeout and action for all stuck states | |
| Both states, different timeouts + different actions | 5 min for PROCESSING (auto-cancel), 15 min for POSTING (alert only) | ✓ |

**User's choice:** "just pick what the best practise in bank" — Claude selected the two-rule approach: different timeouts and different actions because POSTING hangs have fundamentally different compensation semantics (money is already balanced in ledger).

**Notes:** Auto-cancel is safe for PAYMENT_PROCESSING (account-service ACID TX may not have committed). Auto-cancel is NOT safe for POSTING (ledger has written entries; auto-reversing would break double-entry accounting invariants). POSTING timeouts → FAILED + manual review alert.

**User's choice:** "just pick what the best practise in bank" — Claude selected Testcontainers with mocked CDC (direct Kafka publish bypasses Debezium). Debezium integration is validated at deployment time via `scripts/smoke-test-obs.sh`.

**Notes:** Running full Debezium in a JUnit IT adds significant complexity (Debezium Connect container, connector registration, WAL lag) for marginal benefit over direct Kafka publishes in tests. Debezium's correctness is well-tested upstream; what needs IT coverage is the Saga state machine correctness.
