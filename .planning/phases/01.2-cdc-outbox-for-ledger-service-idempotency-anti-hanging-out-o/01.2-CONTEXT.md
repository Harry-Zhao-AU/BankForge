# Phase 01.2: CDC + Outbox for Ledger Service, Idempotency / Anti-Hanging / Compensation — Context

**Gathered:** 2026-04-16
**Status:** Ready for planning

<domain>
## Phase Boundary

Delivers the remaining resilience infrastructure for the transfer Saga. The quick tasks from 2026-04-15 patched the immediate bugs (`@Transactional`, idempotency guard, EOS V2). This phase formalises and extends those patterns:

1. **CDC + Outbox for ledger-service** — replace direct `kafkaTemplate.send()` with the outbox pattern (same approach as account-service). Ledger-service writes an outbox row in the same JPA TX as the ledger entries; Debezium publishes to `banking.transfer.confirmed`.
2. **DLT for ledger-service** — add `DeadLetterPublishingRecoverer` to the `DefaultErrorHandler` so exhausted-retry messages are captured, not silently dropped.
3. **transactionId propagation** — surface `transferId` as a first-class observable ID across Kafka headers, OTel baggage, and HTTP response headers.
4. **Hung-transfer detection** — `@Scheduled` job in payment-service watching `PAYMENT_PROCESSING` (5 min) and `POSTING` (15 min) stuck transfers, with appropriate per-state compensation.
5. **End-to-end integration test** — `TransferSagaIT` covering the async Saga leg (account → ledger → notification) using Testcontainers.

No new services. No Kubernetes. Compose-only.

</domain>

<decisions>
## Implementation Decisions

### CDC + Outbox for ledger-service

- **D-01:** Ledger-service adopts the outbox pattern for publishing `banking.transfer.confirmed`. Same outbox table schema as `account-service.outbox_event` (`aggregatetype`, `aggregateid`, `type`, `payload`, `created_at`) — identical column names so the Debezium `EventRouter` SMT config can be reused.
- **D-02:** `aggregateid = transferId` (becomes the Kafka record key). `aggregatetype = "transfer-confirmation"`. Debezium connector route pattern maps this to topic `banking.transfer.confirmed`.
- **D-03:** New Flyway migration `V3__add_ledger_outbox.sql` creates the `ledger_outbox_event` table. New `LedgerOutboxEvent` JPA entity (same structure as `account-service.OutboxEvent`).
- **D-04:** Modify `LedgerEventListener.onTransferEvent`: write debit entry + credit entry + outbox row in a single `@Transactional` JPA TX. Remove `kafkaTemplate.send()` entirely from the listener — Debezium handles the publish.
- **D-05:** Idempotency replay behaviour: `existsByTransferId` returns true → skip all writes including outbox row → return immediately. This is a true no-op: Debezium already published the original outbox row. No direct Kafka publish needed on replay.
- **D-06:** Remove `KafkaTransactionManager` and `EOSMode.V2` from ledger-service `KafkaConfig` — no longer needed since ledger-service has no direct Kafka producer path. The `@Primary JpaTransactionManager` declaration remains (still needed for Spring Data JPA).
- **D-07:** Retain a non-transactional `KafkaTemplate<String, String>` bean in ledger-service `KafkaConfig` — used exclusively by `DeadLetterPublishingRecoverer` (see D-09).
- **D-08:** New file `infra/debezium/ledger-outbox-connector.json`: Debezium PostgreSQL connector pointing to `ledger-db:5432` (host `ledger-db`, port `5432` from within Compose), watching `public.ledger_outbox_event`, same `EventRouter` SMT configuration, `slot.name = "bankforge_ledger_outbox_slot"`, `publication.name = "bankforge_ledger_outbox_pub"`, `slot.drop.on.stop = true`.
- **D-09:** Update `register-connector.sh` to idempotently register both `bankforge-outbox-connector` (existing, account-service) and `bankforge-ledger-outbox-connector` (new, ledger-service). Same PUT-if-exists / POST-if-new logic.

### DLT for ledger-service

- **D-10:** Add `DeadLetterPublishingRecoverer` to ledger-service's `DefaultErrorHandler`. The recoverer publishes to `banking.transfer.events.DLT` (Spring Kafka `.DLT` suffix convention).
- **D-11:** Retry policy before DLT: `ExponentialBackOff(initialInterval=1000ms, multiplier=2.0, maxElapsedTime=30000ms)` — consistent with existing config. Approximately 3 attempts before falling to DLT.
- **D-12:** DLT message includes Kafka headers: `kafka_original-topic`, `kafka_original-partition`, `kafka_original-offset`, `kafka_exception-fqcn`, `kafka_exception-message`. Spring Kafka's `DeadLetterPublishingRecoverer` adds these automatically.
- **D-13:** Add a `@KafkaListener(topics = "banking.transfer.events.DLT", groupId = "ledger-service-dlt")` DLT handler in ledger-service that logs a structured `ERROR` entry with `transferId`, original offset, and exception — this is the alerting hook.
- **D-14:** The non-transactional `KafkaTemplate` (D-07) is shared between the `DeadLetterPublishingRecoverer` and any future outbox-bypass publish paths. No separate bean needed.

### transactionId propagation

- **D-15:** `transferId` IS the bank transaction ID — no new UUID. "Introduce transactionId across the entire flow" means surfacing `transferId` as a first-class observable identifier at three layers:
  1. **Kafka header** — `transaction-id: <transferId>` set in the outbox row or via a Kafka `ProducerInterceptor` / header enricher. All consumers extract from header first, fall back to payload `transferId` field for backwards compatibility.
  2. **OTel baggage** — `banking.transaction.id = <transferId>` set as OTel `Baggage` in `LedgerEventListener.onTransferEvent` and in `PaymentService.initiateTransfer` so every downstream span in the distributed trace carries the transaction ID.
  3. **HTTP response header** — `X-Transaction-Id: <transferId>` added to payment-service's `POST /transfer` response by `PaymentController`.
- **D-16:** OTel baggage propagation uses the existing `ContextPropagatingTaskDecorator` infrastructure already present in all services. Set baggage via `Baggage.current().toBuilder().put("banking.transaction.id", transferId.toString()).build()` before the listener body executes.

### Hung-transfer detection

- **D-17:** `@Scheduled(fixedDelay = 60_000)` job class `HungTransferDetector` in `payment-service`. Runs every 60 seconds. Two separate queries, one per stuck state.
- **D-18:** `PAYMENT_PROCESSING` rule: find transfers in `PAYMENT_PROCESSING` state with `createdAt < now() - 5 minutes`. Action: transition `COMPENSATING → CANCELLED` via `TransferStateService.cancel()`; publish `banking.transfer.failed` event with payload `{ "transferId": "...", "reason": "PAYMENT_PROCESSING_TIMEOUT" }`. Safe to auto-cancel — money may not have moved in account-service (ACID TX may have not committed, outbox event may never have fired).
- **D-19:** `POSTING` rule: find transfers in `POSTING` state with `updatedAt < now() - 15 minutes`. Action: transition to `FAILED` with `errorMessage = "LEDGER_CONFIRM_TIMEOUT — manual review required"`. Log structured `ERROR` with `transferId` and duration. **Do NOT auto-cancel or publish a reversal event** — ledger entries may be written and balances are correct; the missing piece is only the confirmation event, not the bookkeeping.
- **D-20:** `TransferRepository` needs two new query methods: `findByStateAndCreatedAtBefore(TransferState, Instant)` and `findByStateAndUpdatedAtBefore(TransferState, Instant)`.
- **D-21:** `banking.transfer.failed` topic: use existing pattern (`autoCreateTopics` via Spring Kafka if not pre-created). `notification-service` and `account-service` may subscribe in a future phase; for now the event is published but no consumer is wired. The topic is created with `@RetryableTopic` configured on any future consumer.

### End-to-end integration test

- **D-22:** New test class `TransferSagaIT` in the `ledger-service` test module (or a dedicated `integration-tests` Maven module if complexity warrants — Claude's discretion).
- **D-23:** Testcontainers stack: `postgres:17-alpine` (account-service DB schema, for `execute_transfer` path) + `apache/kafka:3.9.2` (KRaft-native).
- **D-24:** CDC path is mocked — tests publish directly to `banking.transfer.events` via `KafkaTemplate`, bypassing Debezium. Full CDC path validation is a deployment smoke test (`scripts/smoke-test-obs.sh` extended), not a JUnit IT.
- **D-25:** `TransferSagaIT` covers three scenarios:
  1. Happy path — event received → debit + credit written → outbox row written → confirmation published to `banking.transfer.confirmed`
  2. Idempotency replay — duplicate event delivery → exactly 2 ledger entries (no duplicates), exactly 1 outbox row, single confirmation event
  3. DLT routing — poison pill payload → after retry exhaustion, message lands in `banking.transfer.events.DLT`, ERROR log asserted

### Claude's Discretion
- Whether `LedgerOutboxEvent` is a separate entity class or reuses a shared `OutboxEvent` from `common` — the important thing is the DB column names match Debezium expectations.
- Whether `HungTransferDetector` is a `@Component` with `@Scheduled` or a separate `@Configuration`-backed scheduled task — use the simpler form.
- Exact `route.topic.replacement` expression in `ledger-outbox-connector.json` — must resolve to `banking.transfer.confirmed` for `aggregatetype = "transfer-confirmation"`.
- Whether `TransferSagaIT` lives in `ledger-service` or a new `integration-tests` module — prefer `ledger-service` test module unless the test requires account-service beans, in which case create a minimal `integration-tests` Maven module.

</decisions>

<specifics>
## Specific Ideas

- The outbox table in ledger-service is named `ledger_outbox_event` (not `outbox_event`) to avoid ambiguity when both account-service and ledger-service's DBs are referenced in the same Debezium Connect logs.
- The Debezium connector for ledger-service uses `ledger-db` as the hostname (the Compose service name) and port `5432` (container-internal; Compose network).
- `POSTING` hangs must NOT trigger account-service balance reversal — the ledger has already written balanced entries (debit = credit). The only missing piece is payment-service's state update. Manual review is the correct action.
- OTel baggage for `banking.transaction.id` propagates via `ContextPropagatingTaskDecorator` already present in all services — no additional OTel infrastructure needed.

</specifics>

<deferred>
## Deferred Ideas

- Consuming `banking.transfer.failed` in account-service for automatic balance reversal on PAYMENT_PROCESSING timeouts — deferred to Phase 3+ (requires careful idempotency design on the reversal path).

- `INVESTIGATING` state in the FSM for POSTING-hung transfers — deferred; `FAILED` + `LEDGER_CONFIRM_TIMEOUT` error message is sufficient for operational alerting in this phase.
- notification-service idempotency guard — not needed yet; at-least-once + idempotent notification channels is acceptable for now.
- Notification for `banking.transfer.failed` events in notification-service — deferred to Phase 3+.
