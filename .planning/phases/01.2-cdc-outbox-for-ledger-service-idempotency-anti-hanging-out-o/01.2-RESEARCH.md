### Locked Decisions

**CDC + Outbox for ledger-service**
- D-01: Ledger-service adopts the outbox pattern for publishing `banking.transfer.confirmed`. Same outbox table schema as `account-service.outbox_event` — identical column names so the Debezium `EventRouter` SMT config can be reused.
- D-02: `aggregateid = transferId` (becomes the Kafka record key). `aggregatetype = "transfer-confirmation"`. Debezium connector route pattern maps this to topic `banking.transfer.confirmed`.
- D-03: New Flyway migration `V3__add_ledger_outbox.sql` creates the `ledger_outbox_event` table. New `LedgerOutboxEvent` JPA entity (same structure as `account-service.OutboxEvent`).
- D-04: Modify `LedgerEventListener.onTransferEvent`: write debit entry + credit entry + outbox row in a single `@Transactional` JPA TX. Remove `kafkaTemplate.send()` entirely — Debezium handles the publish.

- D-06: Remove `KafkaTransactionManager` and `EOSMode.V2` from ledger-service `KafkaConfig`. The `@Primary JpaTransactionManager` declaration remains.
- D-07: Retain a non-transactional `KafkaTemplate<String, String>` bean in ledger-service `KafkaConfig` — used exclusively by `DeadLetterPublishingRecoverer`.
- D-08: New file `infra/debezium/ledger-outbox-connector.json`: Debezium PostgreSQL connector pointing to `ledger-db:5432`, watching `public.ledger_outbox_event`, `slot.name = "bankforge_ledger_outbox_slot"`, `publication.name = "bankforge_ledger_outbox_pub"`, `slot.drop.on.stop = true`.
- D-09: Update `register-connector.sh` to idempotently register both `bankforge-outbox-connector` and `bankforge-ledger-outbox-connector`.

**DLT for ledger-service**
- D-10: Add `DeadLetterPublishingRecoverer` to ledger-service's `DefaultErrorHandler`. Topic: `banking.transfer.events.DLT`.
- D-11: Retry policy: `ExponentialBackOff(initialInterval=1000ms, multiplier=2.0, maxElapsedTime=30000ms)`.
- D-12: DLT message includes Spring Kafka standard headers automatically.
- D-13: Add `@KafkaListener(topics = "banking.transfer.events.DLT", groupId = "ledger-service-dlt")` DLT handler that logs a structured ERROR.
- D-14: The non-transactional `KafkaTemplate` (D-07) is shared between the `DeadLetterPublishingRecoverer` and any future paths.

**transactionId propagation**
- D-15: `transferId` IS the bank transaction ID — no new UUID. Surface at: (1) Kafka header `transaction-id: <transferId>`, (2) OTel baggage `banking.transaction.id = <transferId>`, (3) HTTP response header `X-Transaction-Id: <transferId>` in payment-service.

## Summary

Phase 01.2 closes the resilience gap left after the quick tasks of 2026-04-15. The outbox pattern is already proven in account-service; this phase mirrors it in ledger-service. All five work streams are well-scoped and technically straightforward because:

1. **Outbox for ledger-service** is a structural copy of the account-service pattern — same entity shape, same Debezium connector schema, different DB hostname and slot names.
2. **DLT** is a `DeadLetterPublishingRecoverer` bean wired into an updated `DefaultErrorHandler` — a two-line change to `KafkaConfig.java` plus a new `@KafkaListener` handler.
3. **transactionId propagation** uses existing `ContextPropagatingTaskDecorator` beans (present in all four services) and the OTel Baggage API (`Baggage.current().toBuilder().put(...).build().makeCurrent()`).
4. **Hung-transfer detection** requires: (a) adding `FAILED` to `TransferState` enum and a transition in `TransferStateMachine`, (b) adding `@EnableScheduling` to the payment-service application class, (c) two new repository query methods, (d) a new `HungTransferDetector` component. The D-19 `FAILED` state is **not currently in the enum** — this is a required addition.
5. **TransferSagaIT** reuses the established `@Testcontainers` + `apache/kafka:3.9.2` + `postgres:17-alpine` pattern already in `LedgerEventListenerIT`.

**Primary recommendation:** Execute the five work streams as sequential waves. Outbox refactor first (breaks the direct Kafka producer path), then DLT wiring (depends on the KafkaConfig refactor), then transactionId (orthogonal but cleanest after listener stabilises), then hung-transfer detection (common module change requires careful sequencing), then end-to-end IT last.

## Architecture Patterns

### Pattern 1: Outbox Entity Mirroring (ledger-service)

The `LedgerOutboxEvent` entity must be a structural copy of `account-service.OutboxEvent` with the same column names (`aggregatetype`, `aggregateid`, `type`, `payload`, `created_at`). The table name MUST differ (`ledger_outbox_event` not `outbox_event`) to avoid ambiguity in shared Debezium logs.

```java
// Source: account-service/entity/OutboxEvent.java (reference)
@Entity
@Table(name = "ledger_outbox_event")  // Different table name
@Getter @Setter @NoArgsConstructor @AllArgsConstructor @Builder
public class LedgerOutboxEvent {

    @Id
    @GeneratedValue(strategy = GenerationType.UUID)
    private UUID id;

    @Column(nullable = false, length = 255)
    private String aggregatetype;   // MUST match Debezium EventRouter field name exactly

    @Column(nullable = false, length = 255)
    private String aggregateid;     // MUST match Debezium EventRouter field name exactly

    @Column(nullable = false, length = 255)
    private String type;

    @JdbcTypeCode(SqlTypes.JSON)
    @Column(nullable = false, columnDefinition = "jsonb")
    private String payload;

    @Column(name = "created_at", nullable = false, updatable = false)
    private Instant createdAt;

    @PrePersist
    protected void onCreate() { this.createdAt = Instant.now(); }
}
```

### Pattern 2: Refactored LedgerEventListener

The listener goes from: write entries + direct `kafkaTemplate.send()` → write entries + outbox row (all in single JPA TX). The `kafkaTemplate` field is removed from the listener constructor entirely.

Key structural changes:
- Inject `LedgerOutboxEventRepository` instead of `KafkaTemplate`
- The idempotency check on replay is now a true no-op (no direct Kafka publish — Debezium already published the original outbox row)
- `@Transactional` remains — wraps debit entry + credit entry + outbox write as single JPA TX

```java
// Revised onTransferEvent body (conceptual)
@Transactional
public void onTransferEvent(@Payload String payload, ...) {
    // ... parse transferId, fromAccountId, toAccountId, amount

    if (ledgerEntryRepository.existsByTransferId(transferId)) {
        log.info("Idempotency guard: skip all writes for transferId={}", transferId);
        return;  // True no-op: Debezium already published original outbox row
    }

    // Set OTel baggage for trace correlation (D-15, D-16)
    Baggage baggage = Baggage.current().toBuilder()
        .put("banking.transaction.id", transferId.toString())
        .build();

    try (Scope scope = baggage.makeCurrent()) {
        ledgerEntryRepository.save(debitEntry);
        ledgerEntryRepository.save(creditEntry);
        ledgerOutboxEventRepository.save(buildOutboxRow(transferId));
        log.info("Ledger entries + outbox written: transferId={}", transferId);
    }
}
```

[VERIFIED: OpenTelemetry Java API docs — Baggage.current().toBuilder().put().build().makeCurrent()]

### Pattern 3: Refactored KafkaConfig (ledger-service)

Remove `KafkaTransactionManager` and `EOSMode.V2`. Add `DeadLetterPublishingRecoverer` to the `DefaultErrorHandler`.

```java
@Configuration(proxyBeanMethods = false)
public class KafkaConfig {

    @Bean
    @Primary
    public JpaTransactionManager transactionManager(EntityManagerFactory emf) {
        return new JpaTransactionManager(emf);
    }

    // Non-transactional KafkaTemplate for DLT only (D-07)
    // Spring Boot auto-configures a KafkaTemplate bean — no @Bean needed unless
    // you need to ensure it is non-transactional. Since transaction-id-prefix is
    // removed from application.yml, the auto-configured template is non-transactional.

    @Bean
    public ConcurrentKafkaListenerContainerFactory<String, String> kafkaListenerContainerFactory(
            ConsumerFactory<String, String> consumerFactory,
            KafkaTemplate<String, String> kafkaTemplate) {

        DeadLetterPublishingRecoverer recoverer =
            new DeadLetterPublishingRecoverer(kafkaTemplate,
                (r, e) -> new TopicPartition("banking.transfer.events.DLT", r.partition()));

        ExponentialBackOff backOff = new ExponentialBackOff(1000L, 2.0);
        backOff.setMaxElapsedTime(30000L);

        ConcurrentKafkaListenerContainerFactory<String, String> factory =
            new ConcurrentKafkaListenerContainerFactory<>();
        factory.setConsumerFactory(consumerFactory);
        factory.setCommonErrorHandler(new DefaultErrorHandler(recoverer, backOff));
        return factory;
    }
}
```

[VERIFIED: Spring Kafka docs — https://docs.spring.io/spring-kafka/reference/kafka/annotation-error-handling.html]

**Critical: Remove `transaction-id-prefix` from `application.yml`**

After removing `KafkaTransactionManager`, the `spring.kafka.producer.transaction-id-prefix: ledger-tx-` property in `application.yml` MUST also be removed. If left in place, Spring Boot's auto-configuration will create a transactional producer factory, and the auto-configured `KafkaTemplate` will be transactional — causing `DeadLetterPublishingRecoverer` to fail because it requires a non-transactional template when not inside a Kafka TX.

Also remove `spring.kafka.consumer.isolation-level: read_committed` since EOS is gone.

### Pattern 4: Debezium Connector for ledger-outbox

The `route.topic.replacement` for `aggregatetype = "transfer-confirmation"` must resolve to the fixed topic name `banking.transfer.confirmed`. Two approaches work:

**Option A — Literal replacement (recommended for single-aggregate connectors):**
```json
"transforms.outbox.route.topic.replacement": "banking.transfer.confirmed"
```
This hardcodes the target topic regardless of the `aggregatetype` value. Since this connector only watches `ledger_outbox_event` which always has `aggregatetype = "transfer-confirmation"`, this is safe and simple.

**Option B — Pattern-based replacement (used in account-service):**
```json
"transforms.outbox.route.by.field": "aggregatetype",
"transforms.outbox.route.topic.replacement": "banking.${routedByValue}.events"
```
With `aggregatetype = "transfer-confirmation"`, this produces `banking.transfer-confirmation.events` — NOT `banking.transfer.confirmed`. This does NOT match the required topic.

**Decision:** Option A (literal `"banking.transfer.confirmed"`) is required for this connector. [VERIFIED: Debezium EventRouter docs — `${routedByValue}` maps directly to the field value]

### Pattern 5: FAILED State and @EnableScheduling (payment-service)

**Critical gap:** `FAILED` is NOT currently in the `TransferState` enum (verified by reading `common/src/main/java/au/com/bankforge/common/enums/TransferState.java`). D-19 requires transitioning hung POSTING transfers to `FAILED`. This requires:

1. Add `FAILED` to `TransferState` enum.
2. Add a `POSTING + FAIL_WITH_NO_REVERSAL → FAILED` transition in `TransferStateMachine` — OR treat D-19's `FAILED` as a direct DB field update bypassing the state machine (since this is an exceptional operational state, not a normal FSM transition). The latter is simpler but less pure.
3. Add `@EnableScheduling` to payment-service's application class (not present — verified by grep).

**Recommended approach for D-19 POSTING→FAILED:** Add `FAILED` to the enum + add it as a direct force-set on the Transfer entity (not via state machine) with `transfer.setState(TransferState.FAILED)` + save. Document this as an intentional FSM bypass for the hung-transfer edge case. This avoids polluting the FSM with a transition that only the scheduler ever uses.

**Alternative:** Add `FAIL_MANUAL` event to `TransferEvent` and a `POSTING + FAIL_MANUAL → FAILED` transition. Cleaner FSM semantics but more ceremony.

### Pattern 7: @Scheduled Hung-Transfer Detector

```java
@Component
@RequiredArgsConstructor
@Slf4j
public class HungTransferDetector {

    private final TransferRepository transferRepository;
    private final TransferStateService transferStateService;
    private final KafkaTemplate<String, String> kafkaTemplate;

    @Scheduled(fixedDelay = 60_000)
    public void detectHungTransfers() {
        detectHungPaymentProcessing();
        detectHungPosting();
    }

    void detectHungPaymentProcessing() {
        Instant threshold = Instant.now().minus(5, ChronoUnit.MINUTES);
        List<Transfer> hung = transferRepository
            .findByStateAndCreatedAtBefore(TransferState.PAYMENT_PROCESSING, threshold);
        for (Transfer t : hung) {
            log.warn("Hung PAYMENT_PROCESSING transfer: transferId={} age={}",
                t.getId(), Duration.between(t.getCreatedAt(), Instant.now()));
            transferStateService.cancel(t.getId(), "PAYMENT_PROCESSING_TIMEOUT");
            publishTransferFailed(t.getId(), "PAYMENT_PROCESSING_TIMEOUT");
        }
    }

    void detectHungPosting() {
        Instant threshold = Instant.now().minus(15, ChronoUnit.MINUTES);
        List<Transfer> hung = transferRepository
            .findByStateAndUpdatedAtBefore(TransferState.POSTING, threshold);
        for (Transfer t : hung) {
            log.error("Hung POSTING transfer (manual review required): transferId={} age={}",
                t.getId(), Duration.between(t.getUpdatedAt(), Instant.now()));
            // Direct state set — not via FSM (FAILED is an operational exceptional state)
            // See research Pattern 5 for rationale
            t.setState(TransferState.FAILED);
            t.setErrorMessage("LEDGER_CONFIRM_TIMEOUT — manual review required");
            transferRepository.save(t);
        }
    }
}
```

`@EnableScheduling` must be added to `PaymentServiceApplication.java` or a `@Configuration` class.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| DLT header injection | Custom header interceptor | `DeadLetterPublishingRecoverer` | Automatically adds 8+ standard headers including topic, partition, offset, exception FQCN, stack trace |
| Retry exhaustion detection | Manual retry counter | `DefaultErrorHandler` + `ExponentialBackOff` | Handles exponential backoff, jitter, non-retryable exceptions |
| Outbox to Kafka routing | Application-level Kafka publish | Debezium EventRouter SMT | Handles topic routing, key extraction, JSON payload expansion from the outbox row |
| OTel context propagation across threads | ThreadLocal | `ContextPropagatingTaskDecorator` | Already present in all services; propagates OTel context across virtual thread hand-offs |
| PostgreSQL WAL reading | Direct pg_logical slot queries | Debezium Kafka Connect | Handles slot creation, reconnection, at-least-once delivery, Kafka offset tracking |

## Common Pitfalls

### Pitfall 1: `transaction-id-prefix` left in application.yml after removing KafkaTransactionManager

**What goes wrong:** Spring Boot auto-configures a transactional `ProducerFactory` when `spring.kafka.producer.transaction-id-prefix` is set. The resulting `KafkaTemplate` is transactional. `DeadLetterPublishingRecoverer` fails at runtime with an error about not being able to send outside a Kafka transaction.

**Why it happens:** `KafkaTransactionManager` is removed but the auto-config property that triggered transactional producer setup is left in place.

**How to avoid:** Remove `spring.kafka.producer.transaction-id-prefix: ledger-tx-` from `ledger-service/src/main/resources/application.yml` in the same wave as the `KafkaConfig.java` refactor.

**Warning signs:** `TransactionRequiredException` or `IllegalStateException: No transaction is in progress` from `KafkaTemplate` in the DLT recoverer at startup or on first retry.

### Pitfall 2: `isolation-level: read_committed` consumer property breaks non-EOS consumer

**What goes wrong:** After removing EOS (`EOSMode.V2` + `KafkaTransactionManager`), the consumer is no longer using transactional reads. If `isolation-level: read_committed` remains in `application.yml`, non-transactional Kafka messages published by non-transactional producers will never be visible to the consumer — they are treated as "uncommitted" indefinitely.

**Why it happens:** `read_committed` isolation only returns records that are committed by a transactional producer. Non-transactional records are not considered committed. With the EOS path removed, ledger-service will primarily receive events from Debezium (non-transactional), so `read_committed` must be changed to `read_uncommitted`.

**How to avoid:** Remove `spring.kafka.consumer.isolation-level: read_committed` from `application.yml`. The default (`read_uncommitted`) is correct for Debezium-sourced events.

**Warning signs:** `LedgerEventListener` receives zero messages after the refactor; Debezium connector shows `RUNNING` status but no events appear in consumer group.

### Pitfall 3: `LedgerEventListenerIT` still configures transactional producer after refactor

**What goes wrong:** `LedgerEventListenerIT` has `registry.add("spring.kafka.producer.transaction-id-prefix", () -> "test-ledger-tx-")` in `@DynamicPropertySource`. After removing EOS from the real application, the IT still forces a transactional producer — tests pass with EOS semantics that the real app no longer uses.

**How to avoid:** Remove this property registration from the IT. Instead, publish test events using a plain (non-transactional) `kafkaTemplate.send()`. The `TransferSagaIT` should be written from the start without this property.

### Pitfall 4: Debezium EventRouter publishes wrong topic name

**What goes wrong:** Using `banking.${routedByValue}.events` as `route.topic.replacement` with `aggregatetype = "transfer-confirmation"` produces `banking.transfer-confirmation.events`, not `banking.transfer.confirmed`. The `TransferConfirmationListener` in payment-service subscribes to `banking.transfer.confirmed` — no messages are ever received.

**How to avoid:** Use the literal topic name `"banking.transfer.confirmed"` as the `route.topic.replacement` value (see Architecture Pattern 4).

**Warning signs:** `banking.transfer.confirmed` has zero messages after ledger-service processes a transfer. `banking.transfer-confirmation.events` topic appears unexpectedly.

### Pitfall 5: FAILED state not in TransferState enum

**What goes wrong:** D-19 requires `POSTING → FAILED` transition for hung transfers. `FAILED` does not exist in `TransferState`. Calling `transfer.setState(TransferState.FAILED)` fails to compile.

**How to avoid:** Add `FAILED` to the `TransferState` enum in the `common` module BEFORE implementing `HungTransferDetector`. Rebuild `common` and all dependent modules. Since `common` is a shared library, any service that depends on it gets the new enum value automatically via Maven build.

**Warning signs:** Compilation error in `HungTransferDetector` when referencing `TransferState.FAILED`.

### Pitfall 6: @EnableScheduling missing from payment-service

**What goes wrong:** `@Scheduled` annotations are silently ignored if `@EnableScheduling` is not present in any configuration class. `HungTransferDetector.detectHungTransfers()` never runs.

**How to avoid:** Add `@EnableScheduling` to `PaymentServiceApplication.java` or a dedicated `SchedulingConfig` class. Verify with a log message in the scheduled method that appears ~60 seconds after service start.

**Warning signs:** No log messages from `HungTransferDetector` after startup; hung transfers remain in stuck states indefinitely.

### Pitfall 7: DLT topic suffix `.DLT` vs `.dlt` vs `-dlt` inconsistency

**What goes wrong:** D-10 specifies topic `banking.transfer.events.DLT` (uppercase `.DLT` suffix). `TransferConfirmationListener` in payment-service already uses `dltTopicSuffix = "-dlt"` via `@RetryableTopic`, producing `banking.transfer.confirmed-dlt`. Using inconsistent suffix conventions creates confusion in monitoring.

**How to avoid:** For ledger-service DLT, hardcode the destination topic in `DeadLetterPublishingRecoverer` using the `BiFunction<ConsumerRecord, Exception, TopicPartition>` constructor to explicitly specify `banking.transfer.events.DLT`. Do not rely on default suffix generation. Document the naming convention.

**Warning signs:** Messages end up in an unexpected topic; DLT handler `@KafkaListener` receives zero messages.

## Open Questions

1. **POSTING→FAILED state machine approach**
   - What we know: `FAILED` is not in `TransferState`. D-19 requires it.
   - What's unclear: Whether to add a FSM transition (`POSTING + FAIL_MANUAL → FAILED`) or bypass the state machine for the scheduler's direct write.
   - Recommendation: Direct write (`transfer.setState(TransferState.FAILED)`) with a comment explaining FSM bypass. This keeps the FSM clean for the normal transfer lifecycle and avoids adding a transition that only one edge-case path uses.

2. **`LedgerEventListenerIT` update scope**
   - What we know: The IT currently tests the direct `kafkaTemplate.send()` path (now being removed) with `read_committed` consumer isolation and transactional producer properties.
   - What's unclear: Whether to update `LedgerEventListenerIT` in-place or deprecate it in favour of `TransferSagaIT`.
   - Recommendation: Update `LedgerEventListenerIT` to remove EOS properties and add an assertion that the outbox row is written. Keep it as the unit-level listener test. `TransferSagaIT` covers the full saga scenario.

3. **`debezium-connect` depends_on in compose.yml**
   - What we know: `debezium-connect` currently only has `depends_on: kafka, account-db`. `ledger-db` is not listed.
   - What's unclear: Whether this matters functionally (Debezium registers connectors lazily after startup).
   - Recommendation: Add `ledger-db: condition: service_healthy` to `debezium-connect.depends_on` for correctness. The connector registration script already handles timing, but the compose dependency makes startup order explicit.
