## What Was Built

### TransferSagaIT (new file)

End-to-end integration test class with `@Testcontainers` + `@SpringBootTest` using:
- `postgres:17-alpine` (db=ledgerdb, user=ledger, pw=secret)
- `apache/kafka:3.9.2` (KRaft mode)

**No EOS properties** in `@DynamicPropertySource` — non-transactional KafkaTemplate matches real application config after Plan 02 cleanup.

Three test scenarios (D-22, D-23, D-24, D-25):

1. **`shouldWriteEntriesAndOutboxOnTransferEvent`** — Sends a valid transfer event; awaits exactly 2 ledger entries (DEBIT + CREDIT); then asserts exactly 1 outbox row with `aggregatetype=transfer-confirmation` and `type=TransferConfirmed` containing the transferId.

2. **`shouldBeIdempotentOnDuplicateDelivery`** — Sends the same transferId twice; awaits exactly 2 ledger entries (not 4); asserts exactly 1 outbox row (not 2). Proves the `existsByTransferId` idempotency guard is a true no-op on replay.

3. **`shouldRoutePoisonPillToDlt`** — Sends an invalid JSON payload; creates a raw `KafkaConsumer` subscribed to `banking.transfer.events.DLT`; awaits up to 60s for the message to appear after retry exhaustion. ExponentialBackOff(1000, 2.0) maxElapsed=30s routes to DLT after ~3 retries.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 - Missing critical detail] Unique poison key per test run**
- **Found during:** Task 1 — DLT test scenario design
- **Issue:** Plan spec used literal `"poison-key"` as the Kafka message key. If the test runs multiple times in the same JVM session, a DLT consumer with `auto.offset.reset=earliest` in a new group would still receive the message — however, using a static key could cause interference if tests share state unexpectedly.
- **Fix:** Used `"poison-key-" + UUID.randomUUID()` as the key to ensure isolation across repeated test runs and parallel execution. The DLT consumer polls for this specific key.
- **Files modified:** TransferSagaIT.java
- **Commit:** f9dabd4
