```java
@Service
@RequiredArgsConstructor
public class IdempotencyService {
    private final StringRedisTemplate redisTemplate;
    private static final Duration TTL = Duration.ofHours(24);
    private static final String PREFIX = "idempotency:";

    public Optional<String> getCached(String key) {
        return Optional.ofNullable(
            redisTemplate.opsForValue().get(PREFIX + key));
    }

    /**
     * Returns true if key was newly set (first request), false if already exists.
     */
    public boolean setIfNew(String key, String responseJson) {
        return Boolean.TRUE.equals(
            redisTemplate.opsForValue().setIfAbsent(
                PREFIX + key, responseJson, TTL));
    }
}
```
Per TXNS-05: Uses `setIfAbsent` with 24h TTL. Lettuce is the default driver (bundled in Spring Boot 4) per critical decision 6. Do NOT add Jedis.

**PaymentService.java** (package `au.com.bankforge.payment.service`):

This is the orchestration service per D-02 and D-04. It owns the transfer lifecycle.

**CRITICAL: Jackson 3 import required.** Use `tools.jackson.databind.ObjectMapper` (Jackson 3, bundled with Spring Boot 4.0.5). Do NOT import `com.fasterxml.jackson.databind.ObjectMapper` — that is the Jackson 2 package which will not compile under Spring Boot 4.

```java
@Service
@RequiredArgsConstructor
@Slf4j
public class PaymentService {
    private final TransferRepository transferRepository;
    private final TransferStateMachine stateMachine;
    private final IdempotencyService idempotencyService;
    private final AccountServiceClient accountServiceClient;
    private final ObjectMapper objectMapper;  // tools.jackson.databind.ObjectMapper

    @Transactional
    public InitiateTransferResponse initiateTransfer(InitiateTransferRequest request) {
        // Step 1: Idempotency check (Redis) per TXNS-05
        Optional<String> cached = idempotencyService.getCached(request.idempotencyKey());
        if (cached.isPresent()) {
            log.info("Duplicate request detected for idempotency key: {}", request.idempotencyKey());
            return deserialize(cached.get());
        }

        // Step 2: Create transfer record in PENDING state
        Transfer transfer = Transfer.builder()
            .fromAccountId(request.fromAccountId())
            .toAccountId(request.toAccountId())
            .amount(request.amount())
            .idempotencyKey(request.idempotencyKey())
            .description(request.description())
            .state(TransferState.PENDING)
            .build();
        transfer = transferRepository.save(transfer);

        try {
            // Step 3: PENDING -> PAYMENT_PROCESSING
            transfer.setState(stateMachine.transition(transfer.getState(), TransferEvent.INITIATE));
            transfer = transferRepository.save(transfer);

            // Step 4: Call account-service for ACID transfer execution
            accountServiceClient.executeTransfer(
                request.fromAccountId(), request.toAccountId(), request.amount(), request.description());

            // Step 5: PAYMENT_PROCESSING -> PAYMENT_DONE
            transfer.setState(stateMachine.transition(transfer.getState(), TransferEvent.PAYMENT_COMPLETE));
            transfer = transferRepository.save(transfer);

            // Step 6: PAYMENT_DONE -> POSTING (ledger-service will handle in Phase 1.1)
            transfer.setState(stateMachine.transition(transfer.getState(), TransferEvent.POST));
            transfer = transferRepository.save(transfer);

            // Step 7: POSTING -> CONFIRMED (in Phase 1.1, this happens after ledger confirms)
            // For Phase 1, we auto-confirm since ledger-service is a stub
            transfer.setState(stateMachine.transition(transfer.getState(), TransferEvent.CONFIRM));
            transfer = transferRepository.save(transfer);

        } catch (Exception e) {
            log.error("Transfer failed for {}: {}", transfer.getId(), e.getMessage());
            // Compensation path: -> COMPENSATING -> CANCELLED
            // NOTE: TransferStateMachine supports FAIL from PAYMENT_PROCESSING, PAYMENT_DONE, and POSTING states.
            // This means the catch block works correctly regardless of which step threw the exception.
            try {
                transfer.setState(stateMachine.transition(transfer.getState(), TransferEvent.FAIL));
                transfer = transferRepository.save(transfer);
                transfer.setState(stateMachine.transition(transfer.getState(), TransferEvent.COMPENSATE));
                transfer.setErrorMessage(e.getMessage());
                transfer = transferRepository.save(transfer);
            } catch (Exception compensationError) {
                log.error("Compensation failed for {}: {}", transfer.getId(), compensationError.getMessage());
                transfer.setErrorMessage("Compensation failed: " + compensationError.getMessage());
                transfer = transferRepository.save(transfer);
            }
        }

        // Step 8: Cache response in Redis for idempotency
        InitiateTransferResponse response = new InitiateTransferResponse(
            transfer.getId(), transfer.getFromAccountId(), transfer.getToAccountId(),
            transfer.getAmount(), transfer.getState().name(), transfer.getCreatedAt());
        idempotencyService.setIfNew(request.idempotencyKey(), serialize(response));

        return response;
    }

    @Transactional(readOnly = true)
    public TransferStatusResponse getTransferStatus(UUID transferId) {
        Transfer transfer = transferRepository.findById(transferId)
            .orElseThrow(() -> new RuntimeException("Transfer not found: " + transferId));
        return new TransferStatusResponse(
            transfer.getId(), transfer.getState().name(), transfer.getAmount(),
            transfer.getFromAccountId(), transfer.getToAccountId(),
            transfer.getErrorMessage(), transfer.getCreatedAt(), transfer.getUpdatedAt());
    }

    /**
     * Returns true if this is a new transfer (not idempotent replay).
     * Used by PaymentController to determine HTTP status code (201 vs 200).
     */
    public boolean isNewTransfer(String idempotencyKey) {
        return idempotencyService.getCached(idempotencyKey).isEmpty();
    }

    private String serialize(InitiateTransferResponse response) {
        try { return objectMapper.writeValueAsString(response); }
        catch (Exception e) { throw new RuntimeException("Serialization failed", e); }
    }

    private InitiateTransferResponse deserialize(String json) {
        try { return objectMapper.readValue(json, InitiateTransferResponse.class); }
        catch (Exception e) { throw new RuntimeException("Deserialization failed", e); }
    }
}
```

Key design points:
- Per D-02: payment-service is the external entry point. All transfers come through here.
- Per D-04: payment-service owns TXNS-04 (state machine) and TXNS-05 (idempotency).
- Per D-05: Uses `TransferStateMachine.transition()` from common module (enum FSM, not Spring State Machine).
- Per D-06: State transitions follow the exact path: PENDING -> PAYMENT_PROCESSING -> PAYMENT_DONE -> POSTING -> CONFIRMED (happy path) or -> COMPENSATING -> CANCELLED (failure path).
- For Phase 1 only: auto-confirms after POSTING since ledger-service is a stub. Phase 1.1 will make confirmation event-driven.
- NOTE: The `@Transactional` on `initiateTransfer` covers the payment-service DB operations (transfer record state updates). The account-service ACID transfer is a separate transaction in account-service's DB — this is intentional (each service owns its own DB per project constraint).
- The catch block applies `TransferEvent.FAIL` to whatever the current state is. The state machine supports FAIL from PAYMENT_PROCESSING, PAYMENT_DONE, and POSTING — so the compensation path works correctly regardless of where the exception occurred.

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-1-06 | Spoofing | Idempotency key | mitigate | Client-provided idempotency key is stored in Redis with 24h TTL. Unique index on transfers.idempotency_key prevents DB-level duplicates even if Redis is unavailable. |
| T-1-07 | Tampering | Transfer state | mitigate | State transitions enforced by TransferStateMachine — only valid transitions allowed. Invalid transitions throw InvalidStateTransitionException. State persisted as @Enumerated(EnumType.STRING) — cannot store arbitrary strings. |
| T-1-08 | Input Validation | InitiateTransferRequest | mitigate | Jakarta @Valid on all fields. @NotNull for account IDs. @DecimalMin("0.01") on amount. @NotBlank on idempotencyKey. GlobalExceptionHandler returns 400 for validation failures. |
| T-1-09 | Denial of Service | Redis idempotency | accept | Redis has no auth in Phase 1 (local dev only). Rate limiting added in Phase 3 via Kong. |
| T-1-10 | Information Disclosure | Error responses | mitigate | GlobalExceptionHandler returns generic error messages. Account-service error details logged server-side, not leaked to client. |
