## Summary

Phase 02.1 adds a GitHub Actions CI pipeline to the BankForge Maven multi-module project. The pipeline must solve four concrete problems: (1) removing a Windows-only `DOCKER_HOST` Surefire env var that will break Testcontainers on ubuntu-latest, (2) running both `*Test.java` and `*IT.java` classes via Surefire's `<includes>` already configured in every service POM, (3) ensuring the `common` module is installed into `.m2` before the 4 parallel service jobs run, and (4) pushing images to ghcr.io with GITHUB_TOKEN.

The fix for `DOCKER_HOST` is a Maven OS-conditional profile: a `<profile>` activated by `<os><family>Windows</family></os>` sets `npipe:////./pipe/docker_engine` and `TESTCONTAINERS_RYUK_DISABLED=true`, while the base Surefire config (without those env vars) applies on Linux. On ubuntu-latest, Testcontainers auto-detects the Docker socket at `/var/run/docker.sock` without any configuration.

Integration tests (`*IT.java`) are already included in Surefire via `<includes>` in every service POM — `mvn test` (not `mvn verify`) runs all tests. The correct CI Maven command is `mvn -B test` per service. The `common` module uses no Surefire Testcontainers config — its tests are pure unit tests and will run fine with the default Surefire behaviour.

**Primary recommendation:** One prerequisite CI job builds and caches the common module (`mvn -B install -pl common -am -Dmaven.test.skip=true`), followed by a matrix of 4 parallel jobs that each run `mvn -B test` then (on main) `docker build` + `docker push`.

## Architecture Patterns

### Workflow Structure

```
.github/
└── workflows/
    └── ci.yml          ← Single workflow file covering all services
```

```
Modules:
common/                 ← Library JAR, no spring-boot-maven-plugin, no Testcontainers
account-service/        ← Depends on common; Testcontainers PostgreSQL
payment-service/        ← Depends on common; Testcontainers PostgreSQL + Redis + Kafka
ledger-service/         ← Depends on common; Testcontainers PostgreSQL + Kafka
notification-service/   ← Depends on common; Testcontainers Kafka
```

### Pattern 1: Prerequisite Job + Matrix

**What:** A `build-common` job installs the `common` module to `.m2` and saves the cache. A `build-and-test` matrix job depends on (`needs: build-common`) and restores the same cache, then builds and tests each service.

**When to use:** Always for multi-module Maven projects with a shared library module. The matrix jobs cannot run until the shared artifact is in the local repository.

**Example:**
```yaml
# Source: GitHub Actions docs + Maven multi-module conventions [CITED: docs.github.com]
jobs:
  build-common:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-java@v4
        with:
          java-version: '21'
          distribution: 'temurin'
          cache: 'maven'
      - name: Install common module
        run: mvn -B install -pl common -am -Dmaven.test.skip=true

  build-and-test:
    needs: build-common
    runs-on: ubuntu-latest
    strategy:
      matrix:
        service: [account-service, payment-service, ledger-service, notification-service]
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-java@v4
        with:
          java-version: '21'
          distribution: 'temurin'
          cache: 'maven'
      - name: Test ${{ matrix.service }}
        run: mvn -B test -pl ${{ matrix.service }}
```

**Why NOT Dockerfile-only approach:** The Dockerfile multi-stage build cannot run the test suite (tests require a live Docker daemon for Testcontainers; the Docker-in-Docker socket is not available inside `docker build`). Tests must run in the GitHub Actions job itself, not inside the Dockerfile builder stage.

### Pattern 2: Conditional Image Push (main-branch only)

**What:** The matrix job conditionally pushes images only on pushes to `main`. PRs skip the push steps.

**Example:**
```yaml
# Source: GitHub Actions docs [CITED: docs.github.com/actions/publishing-packages/publishing-docker-images]
- name: Push image
  if: github.event_name == 'push' && github.ref == 'refs/heads/main'
  uses: docker/build-push-action@v5
  with:
    context: ${{ matrix.service }}
    push: true
    tags: ${{ steps.meta.outputs.tags }}
    labels: ${{ steps.meta.outputs.labels }}
```

### Pattern 3: ghcr.io Auth + Tagging

**What:** docker/login-action with GITHUB_TOKEN + docker/metadata-action for tag generation.

**Example:**
```yaml
# Source: GitHub Docs [CITED: docs.github.com/en/actions/publishing-packages/publishing-docker-images]
permissions:
  contents: read
  packages: write

steps:
  - name: Log in to ghcr.io
    if: github.event_name == 'push' && github.ref == 'refs/heads/main'
    uses: docker/login-action@v3
    with:
      registry: ghcr.io
      username: ${{ github.actor }}
      password: ${{ secrets.GITHUB_TOKEN }}

  - name: Extract metadata
    if: github.event_name == 'push' && github.ref == 'refs/heads/main'
    id: meta
    uses: docker/metadata-action@v5
    with:
      images: ghcr.io/${{ github.repository_owner }}/bankforge-${{ matrix.service }}
      tags: |
        type=raw,value=latest
        type=sha,format=short
```

The `type=sha,format=short` tag produces `sha-<7chars>`. The image tag used in Phase 3 K8s manifests should reference the `sha-<7chars>` form (e.g., `sha-860c190`), not just the raw SHA, because that is what `docker/metadata-action` outputs.

### Pattern 4: Surefire OS-Conditional Profile (pom.xml fix)

**What:** Move the Windows-only Surefire env vars into a Maven profile activated by `<os><family>Windows</family></os>`. The base Surefire config retains `<includes>` for `*Test.java` and `*IT.java` but no `DOCKER_HOST`.

**Example for each service pom.xml:**
```xml
<!-- Source: Apache Maven Profiles docs [CITED: maven.apache.org/guides/introduction/introduction-to-profiles.html] -->
<build>
  <plugins>
    <plugin>
      <groupId>org.apache.maven.plugins</groupId>
      <artifactId>maven-surefire-plugin</artifactId>
      <configuration>
        <includes>
          <include>**/*Test.java</include>
          <include>**/*IT.java</include>
        </includes>
        <!-- NO DOCKER_HOST here — Linux/CI gets auto-detection -->
      </configuration>
    </plugin>
  </plugins>
</build>

<profiles>
  <profile>
    <id>podman-windows</id>
    <activation>
      <os>
        <family>Windows</family>
      </os>
    </activation>
    <build>
      <plugins>
        <plugin>
          <groupId>org.apache.maven.plugins</groupId>
          <artifactId>maven-surefire-plugin</artifactId>
          <configuration>
            <environmentVariables>
              <DOCKER_HOST>npipe:////./pipe/docker_engine</DOCKER_HOST>
              <TESTCONTAINERS_RYUK_DISABLED>true</TESTCONTAINERS_RYUK_DISABLED>
            </environmentVariables>
          </configuration>
        </plugin>
      </plugins>
    </build>
  </profile>
</profiles>
```

**Critical:** Maven profile `<configuration>` in a plugin **merges** with (does not replace) the base plugin configuration. The `<includes>` in the base config are preserved when the Windows profile activates. [VERIFIED: Maven docs behavior — profile plugin config merges with default config]

### Pattern 5: Multi-Stage Dockerfile for a common-dependent Service

**What:** The Dockerfile copies the full repo root and uses `-pl <service> -am` to build only that service and its dependencies (including common). Maven is not available inside the runtime image.

**Example:**
```dockerfile
# Source: Spring Boot multi-stage Dockerfile conventions [ASSUMED — standard pattern]
# Build stage
FROM eclipse-temurin:21-jdk-jammy AS builder
WORKDIR /build

# Copy POMs first for layer caching
COPY pom.xml .
COPY common/pom.xml common/
COPY account-service/pom.xml account-service/

# Download dependencies (cache layer — invalidated only if pom.xml changes)
RUN --mount=type=cache,target=/root/.m2 \
    mvn -B dependency:go-offline -pl account-service -am -Dmaven.test.skip=true

# Copy source
COPY common/src common/src
COPY account-service/src account-service/src

# Build JAR
RUN --mount=type=cache,target=/root/.m2 \
    mvn -B package -pl account-service -am -Dmaven.test.skip=true

# Runtime stage
FROM eclipse-temurin:21-jre-jammy AS runtime
WORKDIR /app

# Create non-root user
RUN addgroup --system appgroup && adduser --system --ingroup appgroup appuser

COPY --from=builder /build/account-service/target/*.jar app.jar
RUN chown appuser:appgroup app.jar

USER appuser
EXPOSE 8080

ENTRYPOINT ["java", "-XX:MaxRAMPercentage=75.0", "-jar", "app.jar"]
```

**Note on `--mount=type=cache`:** This Docker BuildKit cache mount keeps Maven deps between local `docker build` runs but does NOT persist across GitHub Actions jobs (each job gets a fresh runner). The `actions/setup-java` Maven cache handles CI dep caching instead.

### Anti-Patterns to Avoid

- **Running `mvn test` inside Dockerfile builder stage:** Docker-in-Docker is not available; Testcontainers will fail trying to start containers inside a `docker build` context.
- **Single job for all services:** Serial build takes 4x longer; defeats the parallelism goal.
- **Hardcoding `DOCKER_HOST` without OS condition:** Breaks on ubuntu-latest where `/var/run/docker.sock` is auto-detected.
- **Using `docker/metadata-action` `type=sha` without noting the `sha-` prefix:** The output tag is `sha-860c190`, not `860c190`. Phase 3 K8s manifests must use the full `sha-` prefixed form.
- **Pushing images from PRs:** Security risk (fork PRs can push to ghcr.io). The `if: github.ref == 'refs/heads/main'` guard is mandatory.
- **No `fail-fast: false` on matrix:** Default `fail-fast: true` cancels all matrix jobs when one fails. Set `fail-fast: false` so a failure in payment-service does not suppress account-service test results.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Docker registry auth | `echo $TOKEN \| docker login ...` in run step | `docker/login-action@v3` | Handles credential masking, re-auth, multi-registry; officially maintained by Docker |
| Image tag generation | `git rev-parse --short HEAD` in bash | `docker/metadata-action@v5` | Handles `type=sha`, `type=raw`, `type=semver` consistently; output referenced as `steps.meta.outputs.tags` |
| Maven dep caching | `actions/cache` with manual key | `actions/setup-java@v4` with `cache: maven` | Built-in; cache key automatically includes all `pom.xml` hashes across the repo |

**Key insight:** Docker Hub official actions (login, metadata, build-push) are the standard for ghcr.io workflows — they work identically for GitHub Container Registry and Docker Hub.

## Critical Finding: DOCKER_HOST Fix

### Current State (VERIFIED from codebase)

All four service POMs have identical Surefire configuration:

```xml
<plugin>
  <groupId>org.apache.maven.plugins</groupId>
  <artifactId>maven-surefire-plugin</artifactId>
  <configuration>
    <includes>
      <include>**/*Test.java</include>
      <include>**/*IT.java</include>
    </includes>
    <environmentVariables>
      <DOCKER_HOST>npipe:////./pipe/docker_engine</DOCKER_HOST>
      <TESTCONTAINERS_RYUK_DISABLED>true</TESTCONTAINERS_RYUK_DISABLED>
    </environmentVariables>
  </configuration>
</plugin>
```

### The Problem

On ubuntu-latest, Testcontainers **auto-detects** Docker at `/var/run/docker.sock`. Setting `DOCKER_HOST=npipe:////./pipe/docker_engine` points Testcontainers at a Windows named pipe path that does not exist — causing Testcontainers to fail finding a Docker environment. [VERIFIED: Testcontainers docs — DOCKER_HOST overrides auto-detection; java.testcontainers.org/supported_docker_environment/]

### The Fix: OS-Conditional Maven Profile

**Approach:** Move `DOCKER_HOST` and `TESTCONTAINERS_RYUK_DISABLED` into a `<profile>` activated only when `<os><family>Windows</family></os>`. The base Surefire config keeps only `<includes>`.

**Why this approach vs alternatives:**

| Approach | Verdict | Reason |
|----------|---------|--------|
| OS-conditional profile in pom.xml | **RECOMMENDED** | Single source of truth; works for all developers on Windows AND for CI on Linux without any workflow changes |
| Set `DOCKER_HOST` override in ci.yml workflow env | Acceptable but redundant | Requires the workflow to carry knowledge that should live in pom.xml; also does not fix `TESTCONTAINERS_RYUK_DISABLED` being set unnecessarily |
| Remove from pom.xml entirely and document as manual step | Breaks Windows local dev | Developers would need to `export DOCKER_HOST=...` before every `mvn test` run |

**Change required in:** `account-service/pom.xml`, `payment-service/pom.xml`, `ledger-service/pom.xml`, `notification-service/pom.xml` — all four.

## Critical Finding: Multi-Module Job Ordering

### Problem

The `common` module is a library JAR that every service depends on:
```xml
<dependency>
    <groupId>au.com.bankforge</groupId>
    <artifactId>common</artifactId>
    <version>0.1.0-SNAPSHOT</version>
</dependency>
```

If a matrix job runs `mvn test -pl account-service` before `common` is installed to `.m2`, Maven will fail with "Could not resolve dependencies" for `au.com.bankforge:common:0.1.0-SNAPSHOT`.

### Solution: Prerequisite Job

```yaml
jobs:
  build-common:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-java@v4
        with:
          java-version: '21'
          distribution: 'temurin'
          cache: 'maven'
      - name: Build and install common
        run: mvn -B install -pl common -am -Dmaven.test.skip=true

  build-test-service:
    needs: build-common      # ← waits for build-common to complete
    runs-on: ubuntu-latest
    strategy:
      fail-fast: false
      matrix:
        service: [account-service, payment-service, ledger-service, notification-service]
```

**Cache sharing between jobs:** `actions/setup-java` with `cache: maven` uses a cache key based on `hashFiles('**/pom.xml')`. Since `build-common` and each matrix job use the same key (same pom.xml contents), the matrix jobs restore the cache saved by `build-common` — which already contains `au.com.bankforge:common:0.1.0-SNAPSHOT` in `.m2`. [VERIFIED: actions/setup-java README — cache key format includes pom.xml hash]

**Important:** Each GitHub Actions job runs on a fresh runner. The `.m2` directory is NOT automatically shared between jobs — it is only shared via the Actions cache mechanism. The `actions/setup-java cache: maven` step handles this automatically (save on post-job, restore on job start).

## Common Pitfalls

### Pitfall 1: DOCKER_HOST Kills Testcontainers on Linux

**What goes wrong:** `mvn test` fails with "Could not find a valid Docker environment" even though Docker is running on the ubuntu-latest runner.
**Why it happens:** `DOCKER_HOST=npipe:////./pipe/docker_engine` is set as a Surefire env var in pom.xml. On Linux, Testcontainers sees this variable and tries to connect to it, overriding its auto-detection of `/var/run/docker.sock`. The named pipe path does not exist on Linux.
**How to avoid:** Use the OS-conditional Maven profile pattern (see Pattern 4 above). The fix must be in pom.xml, not the CI workflow.
**Warning signs:** Test failure message mentions "DockerClientException" or "Could not find a valid Docker environment" despite Docker being present.

### Pitfall 2: common Module Not in .m2 for Matrix Jobs

**What goes wrong:** `mvn test -pl account-service` fails immediately with unresolved dependency `au.com.bankforge:common:0.1.0-SNAPSHOT`.
**Why it happens:** Each matrix job starts with a clean runner. If the prerequisite job's `.m2` cache was not saved and restored, the common JAR is absent.
**How to avoid:** Ensure both `build-common` and `build-test-service` use the same `actions/setup-java` cache configuration. The cache key (`hashFiles('**/pom.xml')`) will match and the restore will succeed.
**Warning signs:** "Could not resolve dependencies for project au.com.bankforge:account-service" in the first seconds of the maven build.

### Pitfall 3: Image Push from Fork PRs

**What goes wrong:** An external contributor's PR triggers the workflow and attempts to push to ghcr.io. GITHUB_TOKEN for fork PRs has read-only permissions on packages — the push will fail with 403.
**Why it happens:** Without a branch condition on the push steps, the push fires for all `push` events and `pull_request` events.
**How to avoid:** Guard all image push steps with `if: github.event_name == 'push' && github.ref == 'refs/heads/main'`.
**Warning signs:** "403 Forbidden" from ghcr.io in workflow logs on PR runs.

### Pitfall 4: Matrix Default fail-fast Hides Failures

**What goes wrong:** payment-service tests fail; GitHub Actions cancels account-service, ledger-service, and notification-service jobs before they complete — masking whether they pass.
**Why it happens:** Default `fail-fast: true` cancels sibling matrix jobs on first failure.
**How to avoid:** Set `fail-fast: false` on the matrix strategy.
**Warning signs:** "This job was cancelled" message on matrix jobs that did not actually fail.

### Pitfall 5: SHA Tag Format Mismatch with Phase 3 K8s

**What goes wrong:** Phase 3 K8s manifests reference `860c190` but the actual image tag in ghcr.io is `sha-860c190` (the `docker/metadata-action` prefix).
**Why it happens:** `docker/metadata-action` with `type=sha` produces `sha-<7chars>`, not the bare SHA.
**How to avoid:** Document the tag format clearly. Phase 3 manifests should use `sha-<7chars>` form. Alternatively, add a `type=raw,value={{sha}}` custom tag without prefix — but this requires extracting SHA manually.
**Recommendation:** Use `type=raw,value=latest` and `type=sha,format=short` (default prefix `sha-`). Accept the `sha-` prefix. Phase 3 will pin to `sha-XXXXXXX` format.

### Pitfall 6: spring-boot-maven-plugin Repackaging in `install`

**What goes wrong:** `mvn install -pl common -am` triggers the `spring-boot-maven-plugin` repackage goal if it were present in `common/pom.xml`, converting the library JAR to an executable fat JAR — making it unusable as a dependency.
**Why it happens:** Spring Boot Maven Plugin repackages JARs by default.
**How to avoid:** The `common/pom.xml` already has NO `spring-boot-maven-plugin` (verified). This pitfall is already avoided by the project structure.
**Warning signs:** Dependent services get "zip file is empty" or "invalid JAR" errors when resolving common.

```dockerfile
FROM eclipse-temurin:21-jdk-jammy AS builder
WORKDIR /build

# Copy parent and dependency POMs for layer caching
COPY pom.xml ./
COPY common/pom.xml common/
COPY account-service/pom.xml account-service/

# Pre-fetch dependencies (layer cached until pom.xml changes)
RUN mvn -B dependency:go-offline -pl account-service -am -Dmaven.test.skip=true

# Copy source
COPY common/src common/src
COPY account-service/src account-service/src

# Package (skip tests — tests run in the CI job, not in docker build)
RUN mvn -B package -pl account-service -am -Dmaven.test.skip=true

# ---- Runtime stage ----
FROM eclipse-temurin:21-jre-jammy AS runtime
WORKDIR /app

RUN addgroup --system appgroup && adduser --system --ingroup appgroup appuser

COPY --from=builder /build/account-service/target/*.jar app.jar
RUN chown appuser:appgroup app.jar

USER appuser
EXPOSE 8080

ENTRYPOINT ["java", "-XX:MaxRAMPercentage=75.0", "-jar", "app.jar"]
```

**Build context:** `docker build -f account-service/Dockerfile .` (repo root as context, so all module source is accessible).

### pom.xml OS Profile Fix (apply to all 4 service POMs)

```xml
<!-- Remove DOCKER_HOST from base Surefire config -->
<plugin>
  <groupId>org.apache.maven.plugins</groupId>
  <artifactId>maven-surefire-plugin</artifactId>
  <configuration>
    <includes>
      <include>**/*Test.java</include>
      <include>**/*IT.java</include>
    </includes>
    <!-- No environmentVariables here — Linux/CI auto-detects Docker -->
  </configuration>
</plugin>

<!-- Windows-only profile for Podman named pipe -->
<profiles>
  <profile>
    <id>podman-windows</id>
    <activation>
      <os>
        <family>Windows</family>
      </os>
    </activation>
    <build>
      <plugins>
        <plugin>
          <groupId>org.apache.maven.plugins</groupId>
          <artifactId>maven-surefire-plugin</artifactId>
          <configuration>
            <environmentVariables>
              <DOCKER_HOST>npipe:////./pipe/docker_engine</DOCKER_HOST>
              <TESTCONTAINERS_RYUK_DISABLED>true</TESTCONTAINERS_RYUK_DISABLED>
            </environmentVariables>
          </configuration>
        </plugin>
      </plugins>
    </build>
  </profile>
</profiles>
```

## Open Questions

1. **eclipse-temurin:21-jre-jammy tag availability**
   - What we know: Eclipse Temurin 21 JRE jammy is documented as available.
   - What's unclear: Exact current digest / whether jammy is still the latest LTS Ubuntu base or noble (24.04) is preferred.
   - Recommendation: Planner should verify `docker pull eclipse-temurin:21-jre-jammy` locally or check hub.docker.com before writing Dockerfiles. Noble (`21-jre-noble`) would be Ubuntu 24.04. Jammy is Ubuntu 22.04. Both are valid LTS; jammy is more battle-tested for JVM workloads.

2. **`docker/metadata-action` SHA tag format for Phase 3 manifests**
   - What we know: `type=sha,format=short` produces `sha-XXXXXXX` (with prefix).
   - What's unclear: Whether Phase 3 plans will need to strip the `sha-` prefix for `kubectl set image` or whether static manifests just reference the full tag.
   - Recommendation: Document in Phase 3 research that image tags in ghcr.io are `sha-XXXXXXX` (not bare SHA).

3. **Maven profile merge behavior for Surefire `<includes>`**
   - What we know: Maven merges plugin configurations from profiles with base configurations by default.
   - What's unclear: Whether the merge applies to `<includes>` list specifically (lists can be overwritten or merged depending on Maven version and configuration).
   - Recommendation: In the Windows profile's Surefire config, explicitly repeat the `<includes>` block alongside the `<environmentVariables>` block to be safe. This is cheap and eliminates the ambiguity.

### Known Threat Patterns for CI/CD Pipeline

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Secret exposure in build logs | Information disclosure | Never use `docker login` with password in shell args; use `docker/login-action` which masks credentials |
| Image push from fork PRs | Elevation of privilege | Guard push steps with `if: github.ref == 'refs/heads/main'` |
| Unpinned action versions | Tampering (supply chain) | Pin to SHA or use versioned tags (v3, v4, v5); full SHA pinning is best practice for production |
| Unscanned base images | Tampering | Out of scope for Phase 02.1; add Trivy or Snyk in a future phase |
| GITHUB_TOKEN with write on all scopes | Elevation of privilege | Declare minimum permissions: `contents: read` + `packages: write` only |

**In-scope for Phase 02.1:**
- Minimum permissions declaration (`contents: read`, `packages: write`)
- GITHUB_TOKEN for auth (no PAT)
- Push guard on `main` branch only

**Deferred (explicit):**
- Image vulnerability scanning (Trivy/Snyk)
- Dependency vulnerability scanning (OWASP Dependency-Check)
- SBOM generation / provenance attestation (supported by `docker/build-push-action` but not required for Phase 02.1)
