# Phase 2.1: CI/CD Foundation — Context

**Gathered:** 2026-04-15
**Status:** Ready for planning

<domain>
## Phase Boundary

A GitHub Actions CI pipeline that builds all 4 Spring Boot services (+ common module) in
parallel, runs the full test suite, and pushes OCI images to ghcr.io — all on GitHub-hosted
Linux runners. No Kubernetes, no deployment automation, no self-hosted runner required.

This phase exists to give Phases 3–5 a working quality gate before service mesh and AI
integration work begins. Every main-branch merge must pass build + integration tests before
its images land in ghcr.io.

</domain>

<decisions>
## Implementation Decisions

### Runner
- **D-01:** Use **GitHub-hosted ubuntu-latest** runners exclusively. No self-hosted runner in
  this phase. Docker is pre-installed on ubuntu-latest — the Podman constraint is Windows-only
  (local dev machine). Podman is not needed in CI.

### Image Build
- **D-02:** Each service has its own **Dockerfile** in its module directory. Standard
  multi-stage Docker build: Maven build stage (JDK 21) + JRE runtime stage. Explicit,
  readable, full layer control.
- **D-03:** Images are built with `docker build`, pushed via `docker push` using the
  GITHUB_TOKEN (auto-provided by GitHub Actions). No Jib, no Cloud Native Buildpacks.
- **D-04:** Image repository pattern: `ghcr.io/<owner>/bankforge-<service>:<tag>` for each
  of the 4 services (account-service, payment-service, ledger-service, notification-service).

### Test Strategy
- **D-05:** Run the **full test suite** (unit + integration tests) in CI. Testcontainers uses
  Docker on ubuntu-latest runners transparently — no code changes required. The existing
  `DOCKER_HOST` Surefire configuration (Podman named pipe) does NOT apply in CI; the runner's
  Docker daemon is used automatically.
- **D-06:** Maven flag: `-Dmaven.test.skip=true` is NOT used in CI test jobs. The test skip
  flag is reserved for CI image-build jobs only (where tests already passed). Researcher must
  confirm the correct Maven Surefire/Failsafe configuration to ensure ITs run on ubuntu-latest.

### Build Parallelism
- **D-07:** **Matrix strategy** — one GitHub Actions job per service running in parallel:
  matrix of `[account-service, payment-service, ledger-service, notification-service]`. Each
  job is responsible for: build → test → (on main) push image.
- **D-08:** The `common` module must be built and installed to the local `.m2` before the
  parallel service jobs run. Researcher must determine the correct dependency structure:
  either a prerequisite job that caches `.m2` after `mvn install -pl common -am`, or a
  Dockerfile approach where each service's multi-stage build handles the full module graph.

### Image Tagging
- **D-09:** Two tags per image on every main-branch push:
  1. `latest` — floating tag, always tracks most recent main-branch build
  2. `<git-sha>` (short 7-char SHA) — immutable, used in Phase 3 K8s manifests
- **D-10:** PRs and non-main branch pushes: run build + tests only. No image push, no tags.
  Image push is a main-branch-only operation.

### GitHub Secrets
- **D-11:** **GITHUB_TOKEN only**. It is auto-provided by GitHub Actions in every workflow
  run and has sufficient permissions to push packages to ghcr.io within the same repository.
  No PATs, no manually configured secrets needed for image push.
- **D-12:** Researcher must confirm `packages: write` is declared in the workflow `permissions`
  block — required for GITHUB_TOKEN to push to ghcr.io.

### CD Scope
- **D-13:** **CI only — no deployment automation.** This phase ends at `docker push`. No
  `kubectl apply`, no manifest updates, no GitOps automation. Deployment to kind cluster is a
  Phase 3 concern.

### Claude's Discretion
- Dockerfile base image selection — Eclipse Temurin 21-jdk-jammy (build stage) +
  eclipse-temurin:21-jre-jammy (runtime stage) is the recommended starting point; researcher
  verifies availability and confirms smaller alternatives (e.g. distroless).
- Exact Maven cache configuration — `actions/setup-java` with `cache: 'maven'` is standard;
  researcher confirms this works correctly with the multi-module layout.
- Workflow trigger configuration — `on: [push, pull_request]` with branch filters as
  appropriate; researcher proposes trigger spec based on the tagging decisions above.
- Workflow file structure — single `ci.yml` vs separate per-service workflows; single file
  with matrix is preferred.
- Branch protection rules — out of scope for this phase; user can configure after CI is
  working.
- Java 21 setup in CI — `actions/setup-java@v4` with `distribution: 'temurin'` and
  `java-version: '21'`.

</decisions>

<specifics>
## Specific Ideas

- **Parallel build was the stated goal** in STATE.md: "build all 4 services in parallel" —
  the matrix job approach directly satisfies this, with independent failure reporting per
  service.
- **Images must be usable in Phase 3 K8s manifests** — the `<git-sha>` immutable tag is the
  ref that Phase 3 Kubernetes deployment YAML files will pin. Researcher must note this
  constraint so the tagging scheme is consistent with what `kubectl set image` or static
  manifests expect.
- **GITHUB_TOKEN + ghcr.io** is zero-config for same-repo packages — no manual PAT rotation,
  no secret management burden. This is the intended path for learning projects.

</specifics>

## Existing Code Insights

### Reusable Assets
- `compose.yml` (repo root) — Existing Compose file is NOT used in CI pipeline directly;
  CI uses GitHub Actions service containers or Testcontainers. The compose.yml is the local
  dev reference.
- `scripts/smoke-test-obs.sh` — Smoke test exists; CI pipeline may reference it for
  post-build health checks (out of scope for this phase, but researcher should note it).

### Established Patterns
- **Maven build convention** — `"C:/Program Files/JetBrains/..." mvn` is Windows-only. In CI,
  `mvn` from `actions/setup-java` is the correct binary. Do not reference IntelliJ paths in
  workflow YAML.
- **Surefire DOCKER_HOST configuration** (Phase 1 P02 decision) — `npipe:////./pipe/docker_engine`
  in pom.xml Surefire plugin env vars. This must be removed or made conditional for CI (the
  value is Windows-only; ubuntu-latest Docker socket is `/var/run/docker.sock`). Researcher
  must flag this as a change required in `pom.xml`.
- **Integration test naming** — Check whether ITs use `*IT.java` suffix (Failsafe) or
  `*Test.java` (Surefire). Researcher must confirm so the correct Maven lifecycle phase
  (`verify` vs `test`) is used in CI.

### Integration Points
- `.github/workflows/ci.yml` — New file to create (directory does not exist yet)
- `<service>/Dockerfile` — New file per service (4 total); none exist today
- `pom.xml` — Surefire `DOCKER_HOST` env var needs a conditional or removal for CI
  compatibility

<deferred>
## Deferred Ideas

- **Self-hosted runner** — Discussed and decided against for this phase. Becomes relevant only
  if CD automation to local kind cluster is added. Deferred to Phase 3+ if needed.
- **CD / auto-deploy to kind** — Explicitly out of scope. Phase 3 introduces Kubernetes;
  deployment automation (GitOps, `kubectl apply`) belongs there.
- **Branch protection rules** — Useful quality gate (require CI to pass before merge), but
  GitHub repo configuration, not code. User can configure after CI is working. Deferred post-
  Phase 2.1 execution.
- **Multi-platform images (arm64 + amd64)** — Not needed for a local kind cluster on x86_64.
  Deferred until Phase 5 or if cloud deployment is added.
- **Per-service change detection** — Only rebuild services whose code changed. Useful for
  large monorepos; overkill for 4 services. Deferred to backlog.

</deferred>
