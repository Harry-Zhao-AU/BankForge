# Phase 01.1: Code Review Report

**Reviewed:** 2026-04-11
**Depth:** standard
**Files Reviewed:** 15 (+ AustracAuditListener.java and AustracAuditListenerTest.java discovered via filesystem scan)
**Status:** issues_found

## Summary

Phase 01.1 delivers three coherent workstreams: the Debezium CDC outbox pipeline, AUSTRAC compliance
threshold audit logging, and the kind + Podman WSL2 spike. The core patterns are correctly implemented —
BigDecimal is used throughout, `@MockitoBean` / `@MockitoSpyBean` is used (not the removed `@MockBean`),
the Outbox atomic write pattern is correct, `SELECT FOR UPDATE` deadlock ordering is well-reasoned, and
all three Kafka consumers have `@RetryableTopic` + `@DltHandler` wired.

Two critical defects will prevent compilation or cause silent runtime failures:

1. `LedgerEventListener` uses the wrong annotation element name (`backoff`) and the wrong import
   (`org.springframework.retry.annotation.Backoff`) for `@RetryableTopic`. Spring Kafka 4.x defines the
   element as `backOff()` returning `org.springframework.kafka.annotation.BackOff`. This will fail at
   compile time (element not found) or silently apply the annotation default (no retries).

2. `LedgerEventListenerIT` uses the legacy `org.testcontainers.containers.KafkaContainer` (defaults to
   ZooKeeper mode) but supplies the `apache/kafka:3.9.2` image (KRaft-only). The container will start but
   the embedded ZooKeeper configuration path will fail because `apache/kafka` does not include ZooKeeper
   binaries, causing the test to hang or error.

Six warnings cover unsafe JSON field access patterns that route to the DLT (acceptable for a learning
sandbox but worth hardening), a connector configuration choice (`slot.drop.on.stop: true`) that would
cause data loss in any re-deployed environment, a relative-path assumption in a script, a hardcoded
Docker named pipe for Testcontainers on a Podman project, a data-loss risk in `kind-smoke-test.sh`
overwriting `containers.conf` unconditionally, and a potentially unreliable negative assertion in
the AUSTRAC below-threshold test.

## Critical Issues

### CR-01: `LedgerEventListener` — wrong `@RetryableTopic` element name and wrong import

**File:** `ledger-service/src/main/java/au/com/bankforge/ledger/kafka/LedgerEventListener.java:13,32`

**Issue:** The annotation uses `backoff = @Backoff(...)` where `@Backoff` is imported from
`org.springframework.retry.annotation.Backoff`. In Spring Kafka 4.x (verified against
`spring-kafka-4.0.4.jar`), `@RetryableTopic` declares a single element `backOff()` returning
`org.springframework.kafka.annotation.BackOff`. There is no `backoff` element on `@RetryableTopic`.

At compile time this is an unknown element on the annotation — the compiler will reject it. If the build
somehow proceeds (e.g., with a spring-retry version on the classpath that defines a `@RetryableTopic`
wrapper), the backoff settings are silently ignored and the default (no delay, 3 attempts) is applied.
`spring-retry` is also absent from every `pom.xml` in the project, meaning the import itself will fail to
resolve.

The notification-service listeners (`NotificationEventListener`, `AustracAuditListener`) use the correct
form `backOff = @BackOff(...)` with `org.springframework.kafka.annotation.BackOff` — use those as the
reference.

**Fix:**
```java
// Remove this import:
// import org.springframework.retry.annotation.Backoff;

// Add this import:
import org.springframework.kafka.annotation.BackOff;

// Change the annotation element from backoff to backOff:
@RetryableTopic(
    attempts = "4",
    backOff = @BackOff(delay = 1000, multiplier = 2.0, maxDelay = 10000),
    dltTopicSuffix = "-dlt",
    autoCreateTopics = "true",
    numPartitions = "1",
    replicationFactor = "1"
)
```

### CR-02: `LedgerEventListenerIT` — old `KafkaContainer` defaults to ZooKeeper mode; incompatible with `apache/kafka:3.9.2`

**File:** `ledger-service/src/test/java/au/com/bankforge/ledger/kafka/LedgerEventListenerIT.java:10,33-36`

**Issue:** The test imports `org.testcontainers.containers.KafkaContainer` (the legacy Confluent-oriented
class). Bytecode inspection of `testcontainers-kafka-1.20.4.jar` confirms `kraftEnabled` defaults to
`false` in the constructor. Without calling `.withKraft()`, the container attempts to configure embedded
ZooKeeper.

The image supplied is `apache/kafka:3.9.2` — the native Apache Kafka image which does not include
ZooKeeper binaries and only supports KRaft mode. The container will start but the ZooKeeper configuration
path inside Testcontainers' startup logic will fail (missing `zookeeper-server-start.sh`), causing the
test to time out or produce a cryptic error.

Additionally, the legacy class requires `.asCompatibleSubstituteFor("confluentinc/cp-kafka")` when using
non-Confluent images, which is present but insufficient to fix the ZooKeeper/KRaft mismatch.

The notification-service tests use `org.testcontainers.kafka.KafkaContainer` (the new API at
`testcontainers.kafka.*`) which accepts the `apache/kafka` image natively and handles KRaft automatically.
Use that class consistently.

**Fix:**
```java
// Remove:
// import org.testcontainers.containers.KafkaContainer;

// Add:
import org.testcontainers.kafka.KafkaContainer;

// Replace the container declaration:
@Container
static KafkaContainer kafka = new KafkaContainer(
        DockerImageName.parse("apache/kafka:3.9.2")
);
// No asCompatibleSubstituteFor() needed with the new API
```

## Warnings

### WR-01: Unsafe `root.get(field).asText()` — NullPointerException on malformed events routes to DLT

**Files:**
- `ledger-service/src/main/java/au/com/bankforge/ledger/kafka/LedgerEventListener.java:51-54`
- `notification-service/src/main/java/au/com/bankforge/notification/kafka/NotificationEventListener.java:46-49`
- `notification-service/src/main/java/au/com/bankforge/notification/kafka/AustracAuditListener.java:68,74-76`

**Issue:** All three listeners call `root.get("fieldName").asText()` without null-checking `root.get()`
first. In Jackson (both 2 and 3), `JsonNode.get(String)` returns `null` if the field is absent (as
opposed to `path()` which returns `MissingNode`). Calling `.asText()` on `null` throws
`NullPointerException`.

For the AUSTRAC listener this is a compliance concern: a malformed event that is missing the `amount`
field silently crashes the listener. `@RetryableTopic` will retry the NPE 4 times and then route to the
DLT. The DLT handler only logs — there is no alerting. A flood of malformed events will fill retry topics
without triggering any human-visible alert beyond a log line.

For a learning sandbox, routing to the DLT on parse failure is acceptable. The risk is that a single
malformed event blocks a partition's processing for the entire retry window (4 attempts × up to 10s
backoff = ~30s per bad message).

**Fix:** Use `path()` instead of `get()`, or check for null/missing explicitly:
```java
// Safe approach using path() + isMissingNode():
JsonNode amountNode = root.path("amount");
if (amountNode.isMissingNode() || amountNode.isNull()) {
    throw new IllegalArgumentException("Transfer event missing required field: amount. payload=" + payload);
}
BigDecimal amount = new BigDecimal(amountNode.asText());
```

### WR-02: `infra/debezium/outbox-connector.json` — `slot.drop.on.stop: true` destroys replication slot on restart

**File:** `infra/debezium/outbox-connector.json:16`

**Issue:** `"slot.drop.on.stop": "true"` instructs Debezium to delete the PostgreSQL replication slot
when the connector stops. If the connector stops for any reason (container restart, `compose down`,
connector re-registration), all WAL changes accumulated since the last committed offset are lost. When
the connector restarts, it starts from the current WAL head — events that occurred during the gap are
never published.

For a local development environment this is intentional (prevents replication slot bloat across
`compose down` cycles), but the value should be documented as dev-only. If this config file is used as a
template for staging/production, it will cause silent message loss.

**Fix:** Add a comment, and consider parameterising the value or using a separate connector config per
environment:
```json
"slot.drop.on.stop": "true",
"_comment_slot_drop": "DEV ONLY — set false in staging/production to prevent WAL gap on connector restart"
```
Note: JSON does not support comments natively; Kafka Connect ignores unknown fields, so a `_comment_*`
key is a workable convention. Alternatively, maintain separate `outbox-connector-dev.json` and
`outbox-connector-prod.json` files.

### WR-03: `register-connector.sh` — relative path to connector JSON breaks if script is not run from repo root

**File:** `scripts/register-connector.sh:17,22`

**Issue:** Lines 17 and 22 reference `infra/debezium/outbox-connector.json` as a relative path. If the
script is executed from any directory other than the repository root (e.g., `cd scripts &&
./register-connector.sh`), `jq` and `curl` will fail to find the file with a misleading error message.
`set -euo pipefail` is present, so the script will abort, but the error (`No such file or directory`) may
not be obvious.

**Fix:** Resolve paths relative to the script's own location:
```bash
SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
CONNECTOR_JSON="$SCRIPT_DIR/../infra/debezium/outbox-connector.json"

# Then replace the two occurrences:
--data "$(jq '.config' "$CONNECTOR_JSON")"
--data @"$CONNECTOR_JSON"
```

### WR-04: `pom.xml` (ledger-service, notification-service) — `DOCKER_HOST` hardcoded to Docker named pipe; incompatible with Podman

**Files:**
- `ledger-service/pom.xml:137`
- `notification-service/pom.xml:120`

**Issue:** The Surefire `environmentVariables` block sets
`DOCKER_HOST=npipe:////./pipe/docker_engine` — the Windows named pipe for Docker Desktop. The project
uses Podman (rootful, WSL2). Testcontainers will attempt to connect to the Docker named pipe when this
environment variable is set, fail to find it (Podman uses a different socket path), and throw
`DockerClientException` before any container starts.

The correct Testcontainers setup for Podman on Windows WSL2 is either:
- Set `DOCKER_HOST=unix:///run/user/0/podman/podman.sock` (rootful Podman socket path in WSL2), or
- Use a `testcontainers.properties` file with `docker.host=...` so it is not hardcoded in the POM.

**Fix:**
```xml
<environmentVariables>
    <!-- Rootful Podman socket in WSL2 — adjust path if your Podman socket differs -->
    <DOCKER_HOST>unix:///run/user/0/podman/podman.sock</DOCKER_HOST>
    <TESTCONTAINERS_RYUK_DISABLED>true</TESTCONTAINERS_RYUK_DISABLED>
</environmentVariables>
```
Or remove the `DOCKER_HOST` override entirely and rely on Testcontainers' auto-detection, which checks
the Podman socket when Docker is absent.

### WR-05: `kind-smoke-test.sh` — unconditional overwrite of `/root/.config/containers/containers.conf`

**File:** `scripts/kind-smoke-test.sh:64-65`

**Issue:** Line 65 unconditionally overwrites `/root/.config/containers/containers.conf` with a
single-key config. Any pre-existing Podman configuration (custom registries, network settings,
cgroup configuration) is silently destroyed each time the script runs. The script header comments describe
this as a one-time manual prerequisite but the active code (lines 59-66) executes it every run.

**Fix:** Check for existence before writing, or use append-only logic:
```bash
CONTAINERS_CONF="/root/.config/containers/containers.conf"
mkdir -p "$(dirname "$CONTAINERS_CONF")"
if ! grep -q 'log_driver' "$CONTAINERS_CONF" 2>/dev/null; then
    printf '[containers]\nlog_driver = "k8s-file"\n' >> "$CONTAINERS_CONF"
    echo "Appended k8s-file log_driver to $CONTAINERS_CONF"
else
    echo "log_driver already configured in $CONTAINERS_CONF — skipping"
fi
```

### WR-06: `AustracAuditListenerTest` — negative assertion may pass before the consumer has processed the message

**File:** `notification-service/src/test/java/au/com/bankforge/notification/kafka/AustracAuditListenerTest.java:140`

**Issue:** The below-threshold test uses:
```java
await().atMost(30, TimeUnit.SECONDS).during(5, TimeUnit.SECONDS).untilAsserted(() ->
    assertThat(found).isFalse()
);
```

Awaitility's `during(5s)` means the condition must hold continuously for 5 seconds. However, polling
begins immediately at `t=0`. If the Kafka consumer has not yet processed the message (e.g., poll cycle
hasn't started), `found` is `false` from `t=0` regardless. The 5-second window starts counting
immediately, and the test passes before the consumer ever runs. The sub-threshold event could be
processed at `t=6s` — after the assertion window closes — producing a false pass.

A more reliable pattern for negative assertions is to first confirm the consumer is alive (e.g., ensure a
sentinel message on a different topic is consumed) before asserting the audit logger remains empty.
Alternatively, accept the limitation and document it:

**Fix:**
```java
// Publish a marker to verify consumer liveness before the negative assertion
// (requires a test-only listener or a counter mechanism)
// Simpler: extend the during window to be longer than the consumer's max poll interval
await().atMost(30, TimeUnit.SECONDS)
       .during(15, TimeUnit.SECONDS)  // > max.poll.interval.ms default (300s) is overkill;
       .untilAsserted(() ->            // 15s gives reasonable confidence for local tests
           assertThat(found).isFalse()
       );
```
Note: the 5-second window is the primary concern. Even extending it is only a probabilistic improvement.
The authoritative fix is a consumer-liveness probe, but for a learning project the extended window is
pragmatic.

### IN-02: `LedgerEventListener` — `@Transactional` on a `@KafkaListener` method requires explicit transaction manager wiring

**File:** `ledger-service/src/main/java/au/com/bankforge/ledger/kafka/LedgerEventListener.java:42`

**Issue:** `@Transactional` on a `@KafkaListener` method creates a JPA transaction that wraps the
`ledgerEntryRepository.save()` calls. This is correct for ensuring atomic double-entry writes. However,
the `@RetryableTopic` retry mechanism creates a new consumer invocation for each retry attempt in a new
thread — `@Transactional` will create a new transaction per invocation, which is the desired behaviour.
The concern is that this pattern is known to interact subtly with Kafka's `ChainedKafkaTransactionManager`
if one is configured later (e.g., when adding exactly-once semantics). No issue today; document the
intent if EOS delivery is added.

**Fix:** Add a comment to record the design intent:
```java
// @Transactional ensures both LedgerEntry rows (debit + credit) are written atomically.
// If exactly-once Kafka semantics are added later, wire a ChainedKafkaTransactionManager
// and ensure the Kafka transaction manager is listed first in the chain.
@Transactional
public void onTransferEvent(...) {
```

### IN-03: `OutboxEventRepository.findTransfersByAccountId` — `accountId` parameter is an unvalidated String in a native query

**File:** `account-service/src/main/java/au/com/bankforge/account/repository/OutboxEventRepository.java:38`

**Issue:** The native JSONB query uses a `@Param` bind parameter (`= :accountId`) which is safe against
SQL injection — Spring Data JPA uses a prepared statement. This is correct. However, the `accountId`
parameter is declared as `String`, not `UUID`. If the calling code passes user-supplied input that is not
a valid UUID (e.g., a path variable from a REST endpoint), PostgreSQL will evaluate the JSONB `->>`
comparison against the malformed string and return an empty result rather than throwing an error, making
malformed IDs invisible to callers. Consider accepting `UUID` and calling `.toString()` internally, or
adding a validation note.

**Fix:**
```java
// Document that callers must supply a valid UUID string:
// @param accountId  Must be a valid UUID string (e.g., from a validated REST path variable)
List<OutboxEvent> findTransfersByAccountId(@Param("accountId") String accountId);
```
Or change the signature to `UUID` and convert in the query:
```java
List<OutboxEvent> findTransfersByAccountId(@Param("accountId") UUID accountId);
// and adjust query: payload->>'fromAccountId' = CAST(:accountId AS text)
```
