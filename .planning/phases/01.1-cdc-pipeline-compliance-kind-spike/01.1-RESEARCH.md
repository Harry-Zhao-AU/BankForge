## User Constraints (from CONTEXT.md)

### Locked Decisions

- **D-01:** Kafka runs in KRaft mode — no ZooKeeper. `KAFKA_PROCESS_ROLES=broker,controller`. Use Kafka 3.8+ image.
- **D-02:** Outbox + Debezium CDC only — no polling outbox, no dual-write. Debezium reads PostgreSQL WAL directly via logical replication.
- **D-03:** Debezium connector config must include `slot.drop.on.stop=true` to prevent replication slot leaks on container restart.
- **D-04:** PostgreSQL `wal_level=logical` and `max_replication_slots=5` already set in Phase 1 — no DB restart needed.
- **D-05:** Outbox EventRouter SMT (`io.debezium.transforms.outbox.EventRouter`) routes outbox rows to correct Kafka topics by event type.
- **D-06:** DLT (Dead Letter Topic) topics configured from Day 1 for all financial Kafka consumers. Silent event loss is unacceptable for financial events. Use Spring Kafka's `DeadLetterPublishingRecoverer` with exponential backoff.
- **D-07:** AUSTRAC threshold = AUD $10,000. Any transfer >= $10,000 produces a structured compliance event with: timestamp, source account ID, destination account ID, amount, transfer ID.
- **D-08:** Compliance events logged as structured JSON to a dedicated audit log (file or separate Kafka topic). Must be queryable for a given transfer ID.
- **D-09:** Spike validates `KIND_EXPERIMENTAL_PROVIDER=podman` single-node cluster startup, test pod reachability, and DNS resolution of cluster-internal names from a pod. This is a prerequisite gate for Phase 3.
- **D-10:** If Podman rootless fails with kind, try rootful Podman. Document the result in STATE.md so Phase 3 planning knows the working mode.

### Claude's Discretion

- Kafka topic naming convention (e.g. `banking.outbox.transfers` vs `transfer-events`) — choose a consistent pattern and document it.
- Compose service startup ordering — PostgreSQL -> Kafka -> Debezium connector registration ordering via `depends_on` + health checks.
- Whether kind spike is a shell script or a Makefile target.

### Deferred Ideas (OUT OF SCOPE)

- Schema Registry (Confluent) — not needed for Phase 1.1; consider for Phase 2+ if event schema evolution becomes a problem.
- Multi-node kind cluster — single-node is sufficient for Phase 3 validation.

## Summary

Phase 1.1 wires three independent infrastructure concerns on top of the working Phase 1 services: (1) the Kafka KRaft broker and Debezium Connect container are added to compose.yml; (2) a Debezium connector is configured to stream outbox_event WAL changes through EventRouter SMT to domain Kafka topics; (3) ledger-service and notification-service gain @KafkaListener beans that consume those topics with dead-letter routing from Day 1; (4) an AUSTRAC threshold detector logs structured audit events for transfers >= $10,000; and (5) a standalone shell script validates that a kind cluster can start via Podman rootful mode with DNS working.

The Phase 1 compose.yml already has all four PostgreSQL databases with `wal_level=logical` and `max_replication_slots=5` set. The outbox_event table (account-service) has the exact column names the Debezium EventRouter SMT expects by default: `id`, `aggregatetype`, `aggregateid`, `type`, `payload`. No schema changes are required for the Debezium setup.

**AUBN-01 status:** BSB validation is ALREADY COMPLETE — `@ValidBsb` annotation applied to `CreateAccountRequest.bsb`, `BsbValidator` (pattern `^\d{3}-\d{3}$`) exists in the `common` module, and tests pass. Phase 1.1 plans should NOT re-implement BSB validation.

**Primary recommendation:** Wire Kafka KRaft + Debezium as compose.yml additions. Use `apache/kafka:3.9.2` (latest stable, KRaft native). Use `quay.io/debezium/connect:3.1` for Kafka Connect. Register the connector via a `scripts/register-connector.sh` script. Use `@RetryableTopic` with exponential backoff for all @KafkaListener methods. The kind spike must run rootful Podman (`podman machine set --rootful`) — rootless is confirmed not supported on Windows/WSL2 with cgroupv2.

### Pattern 1: Kafka KRaft Single-Node in compose.yml

**What:** Single-node Kafka acting as both broker and controller. No ZooKeeper container.
**When to use:** All local dev. `KAFKA_PROCESS_ROLES=broker,controller` on one container.
**Example:**
```yaml
# Source: apache/kafka Docker Hub, Kafka KRaft docs
kafka:
  image: apache/kafka:3.9.2
  container_name: kafka
  environment:
    KAFKA_NODE_ID: 1
    KAFKA_PROCESS_ROLES: broker,controller
    KAFKA_LISTENERS: PLAINTEXT://0.0.0.0:9092,CONTROLLER://0.0.0.0:9093,PLAINTEXT_HOST://0.0.0.0:29092
    KAFKA_ADVERTISED_LISTENERS: PLAINTEXT://kafka:9092,PLAINTEXT_HOST://localhost:29092
    KAFKA_CONTROLLER_LISTENER_NAMES: CONTROLLER
    KAFKA_LISTENER_SECURITY_PROTOCOL_MAP: CONTROLLER:PLAINTEXT,PLAINTEXT:PLAINTEXT,PLAINTEXT_HOST:PLAINTEXT
    KAFKA_CONTROLLER_QUORUM_VOTERS: 1@kafka:9093
    KAFKA_OFFSETS_TOPIC_REPLICATION_FACTOR: 1
    KAFKA_TRANSACTION_STATE_LOG_REPLICATION_FACTOR: 1
    KAFKA_TRANSACTION_STATE_LOG_MIN_ISR: 1
    CLUSTER_ID: MkU3OEVBNTcwNTJENDM2Qk  # base64-encoded UUID; generate with: kafka-storage.sh random-uuid
    KAFKA_AUTO_CREATE_TOPICS_ENABLE: "true"
    KAFKA_LOG_RETENTION_HOURS: 168
  healthcheck:
    test: ["CMD-SHELL", "/opt/kafka/bin/kafka-topics.sh --bootstrap-server kafka:9092 --list"]
    interval: 10s
    timeout: 5s
    retries: 10
    start_period: 20s
  ports:
    - "29092:29092"   # host access for local tools
```

**Two-listener rule:** `PLAINTEXT://kafka:9092` is for container-to-container (Debezium Connect uses this). `PLAINTEXT_HOST://localhost:29092` is for the host machine (Kafka UI, local CLI). Both are required in Podman Compose because the container hostname resolves only within the Podman network. **What:** The Kafka Connect connector configuration that maps outbox_event rows to domain Kafka topics.

    "slot.name": "bankforge_outbox_slot",
    "slot.drop.on.stop": "true",
    "table.include.list": "public.outbox_event",
    "tombstones.on.delete": "false",

    "transforms.outbox.type": "io.debezium.transforms.outbox.EventRouter",

### Pattern 4: Spring Kafka @KafkaListener with @RetryableTopic (DLT)

**What:** Financial Kafka consumers that retry with exponential backoff and route failures to a DLT.
**When to use:** All @KafkaListener methods that process financial events (CORE-04, CORE-05).

```java
// Source: docs.spring.io/spring-kafka/reference/retrytopic/features.html
// Spring Kafka 4.0.4 verified API

@Component
@Slf4j
public class LedgerEventListener {

    @RetryableTopic(
        attempts = "4",                           // 1 original + 3 retries
        backoff = @BackOff(delay = 1000, multiplier = 2.0, maxDelay = 10000),
        dltTopicSuffix = "-dlt",                  // banking.transfer.events-dlt
        autoCreateTopics = "true",
        numPartitions = "1",
        replicationFactor = "1"
    )
    @KafkaListener(
        topics = "banking.transfer.events",
        groupId = "ledger-service",
        containerFactory = "kafkaListenerContainerFactory"
    )
    public void onTransferEvent(
            @Payload String payload,
            @Header(KafkaHeaders.RECEIVED_TOPIC) String topic,
            @Header(KafkaHeaders.RECEIVED_PARTITION) int partition,
            @Header(KafkaHeaders.OFFSET) long offset) {

        // Parse the Debezium EventRouter payload (JSON, no schema envelope)
        // aggregateid is the Kafka message key (transfer UUID)
        // payload is the outbox_event.payload JSONB field content
        log.info("Processing transfer event from topic={} partition={} offset={}", topic, partition, offset);
        // ... process and write to ledger_entries
    }

    @DltHandler
    public void handleDlt(
            @Payload String payload,
            @Header(KafkaHeaders.RECEIVED_TOPIC) String topic) {
        log.error("Transfer event landed in DLT — topic={} payload={}", topic, payload);
        // Emit a metric or alert here for financial DLT events
    }
}
```

**DLT topic name:** `banking.transfer.events-dlt` (Spring Kafka default suffix is `-dlt`). [VERIFIED: Spring Kafka 4.0.4 docs]

**Spring Kafka KafkaListener container factory** needs to be configured in application.yml:
```yaml
spring:
  kafka:
    bootstrap-servers: kafka:9092
    consumer:
      group-id: ledger-service
      auto-offset-reset: earliest
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: org.apache.kafka.common.serialization.StringDeserializer
    producer:
      key-serializer: org.apache.kafka.common.serialization.StringSerializer
      value-serializer: org.apache.kafka.common.serialization.StringSerializer
```

A `KafkaTemplate` bean must exist for the `@RetryableTopic` framework to publish to retry and DLT topics. Spring Boot auto-configures one from the producer properties above.

### Pattern 5: AUSTRAC Threshold Detection

**What:** A Kafka consumer that checks every transfer event for the AUD $10,000 threshold and logs a structured audit event if crossed.
**When to use:** All transfers from the `banking.transfer.events` topic.
**Hook point:** As a separate @KafkaListener in notification-service (separate group ID from the ledger consumer). No new service needed.

```java
// Source: [ASSUMED] — standard Spring @KafkaListener + SLF4J structured logging pattern

@Component
@Slf4j
public class AustracAuditListener {

    private static final BigDecimal THRESHOLD = new BigDecimal("10000.00");
    // Logger must be wired to a dedicated appender or topic — see D-08

    @KafkaListener(
        topics = "banking.transfer.events",
        groupId = "austrac-audit",
        containerFactory = "kafkaListenerContainerFactory"
    )
    public void onTransferEvent(@Payload String payload) {
        // Deserialize payload (Jackson ObjectMapper)
        // Check amount >= THRESHOLD
        // If yes: log structured JSON audit event with:
        //   { "austrac_event": true, "timestamp": ..., "transfer_id": ...,
        //     "from_account_id": ..., "to_account_id": ..., "amount": ..., "currency": "AUD" }
    }
}
```

**Audit log format (D-08):** Structured JSON via Logback, separate logger name `AUSTRAC_AUDIT` routed to a dedicated appender (`austrac-audit.log`) for queryability. Do NOT use the default root logger — mix of operational and compliance logs is a trap.

**logback-spring.xml snippet:**
```xml
<appender name="AUSTRAC_FILE" class="ch.qos.logback.core.rolling.RollingFileAppender">
  <file>logs/austrac-audit.log</file>
  <encoder class="net.logstash.logback.encoder.LogstashEncoder"/>
</appender>
<logger name="AUSTRAC_AUDIT" level="INFO" additivity="false">
  <appender-ref ref="AUSTRAC_FILE"/>
</logger>
```

Use `LoggerFactory.getLogger("AUSTRAC_AUDIT")` in the listener. This keeps AUSTRAC events queryable by `grep transfer_id austrac-audit.log | jq .`. [ASSUMED: Logstash encoder — standard pattern for structured JSON; logstash-logback-encoder is included transitively via Spring Boot 4 when using structured logging]

### Anti-Patterns to Avoid

- **Using bitnami/kafka image instead of apache/kafka:** bitnami uses `KAFKA_CFG_*` prefixed env vars, not the native `KAFKA_*` format. Mixing documentation examples causes startup failures.
- **Registering the Debezium connector at container startup (entrypoint):** The connector must be registered AFTER Kafka Connect is healthy. Use a separate script, not an entrypoint command.
- **Omitting `tombstones.on.delete=false`:** Without this, Debezium sends a null (tombstone) message when an outbox row is deleted, which causes Spring Kafka to throw a `NullPointerException` in the deserializer.
- **Using `publication.autocreate.mode=all_tables`:** This creates a PostgreSQL publication tracking ALL tables. For an outbox-only setup, use `filtered` with `table.include.list=public.outbox_event` to avoid WAL overhead from non-outbox tables.
- **Setting `slot.drop.on.stop=false` in compose dev environment:** Replication slot accumulation causes PostgreSQL WAL disk to grow unboundedly until `max_replication_slots` is exhausted, blocking all future CDC connections.
- **Using `maxAttempts` without accounting for the first attempt:** In Spring Kafka, `maxAttempts=4` means 1 original + 3 retries, not 4 retries. Off-by-one means one fewer retry than intended.
- **Placing AUSTRAC audit events in the root logger:** Compliance logs must be separable from operational logs. A dedicated `AUSTRAC_AUDIT` logger routing to its own file appender is required.
- **Running kind with rootless Podman on Windows:** Rootless Podman does not support cgroupv2 in WSL2 by default. This is a known incompatibility. Only rootful Podman works. [VERIFIED: podman-desktop.io docs, kind issue tracker]

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| WAL-based CDC from PostgreSQL to Kafka | Custom log reader | Debezium PostgreSQL connector | Handles slot management, WAL position tracking, reconnection, schema changes |
| Outbox row routing to topic by aggregate type | Custom Kafka producer SMT | `io.debezium.transforms.outbox.EventRouter` | Built-in SMT; handles field mapping, topic routing, tombstone suppression |
| Consumer retry with exponential backoff | Manual retry loop or `Thread.sleep()` | `@RetryableTopic` + Spring Kafka | Handles retry topic creation, DLT routing, backoff timings, `@DltHandler` |
| Dead letter queue for failed Kafka messages | Custom error handler writing to another topic | `DeadLetterPublishingRecoverer` (auto-wired by `@RetryableTopic`) | Preserves original headers, exception details, offset metadata |
| BSB format validation | Custom regex check in service layer | `@ValidBsb` (already in `common` module) | Already implemented and tested in Phase 1 — do not re-implement |
| KRaft cluster ID generation | Hardcoded string | `kafka-storage.sh random-uuid` (or reuse `MkU3OEVBNTcwNTJENDM2Qk`) | Must be a valid base64-encoded UUID; malformed IDs cause broker startup failure |

**Key insight:** Debezium + EventRouter is a complete outbox-to-Kafka solution. The only application code needed is the connector JSON configuration — no Java code touches the CDC pipeline itself.

## Common Pitfalls

### Pitfall 1: Debezium Replication Slot Leak

**What goes wrong:** Container stops and restarts, Debezium creates a new slot but leaves the old one running. After 5 restarts, PostgreSQL hits `max_replication_slots=5` and refuses new connections.
**Why it happens:** `slot.drop.on.stop` defaults to `false` to preserve WAL offsets for production restarts.
**How to avoid:** Set `slot.drop.on.stop=true` in the connector config (D-03). Add a `make clean` target that runs `SELECT pg_drop_replication_slot('bankforge_outbox_slot')` on account-db.
**Warning signs:** `pg_replication_slots` query returns multiple rows with the same slot name prefix. Debezium log shows `FATAL: all replication slots are in use` on startup.

### Pitfall 2: Kafka Connect Starts Before Kafka Is Ready

**What goes wrong:** Debezium Connect starts, tries to connect to Kafka, fails, and enters a crash loop before Kafka's internal topics are initialised.
**Why it happens:** `depends_on: service_started` only waits for container start, not Kafka being ready to accept connections.
**How to avoid:** Use `depends_on: kafka: condition: service_healthy` in compose.yml. The health check (`kafka-topics.sh --list`) must pass before Debezium Connect starts.
**Warning signs:** Debezium Connect logs `ConnectException: Error while attempting to create/find topic` on first startup.

### Pitfall 3: Kafka Advertised Listener Mismatch in Podman

**What goes wrong:** Debezium Connect (running in the Podman network) cannot reach Kafka because `KAFKA_ADVERTISED_LISTENERS` advertises `localhost` instead of the container hostname `kafka`.
**Why it happens:** `localhost` in a container resolves to that container's loopback, not the Kafka container.
**How to avoid:** Advertise two listeners: `PLAINTEXT://kafka:9092` (for internal container-to-container) and `PLAINTEXT_HOST://localhost:29092` (for host access). Debezium uses `BOOTSTRAP_SERVERS: kafka:9092`.
**Warning signs:** Debezium Connect starts but cannot register any connectors; Kafka logs show no bootstrap requests from the `debezium-connect` container.

### Pitfall 4: EventRouter SMT Routes to Wrong Topic Due to Case Sensitivity

**What goes wrong:** `aggregatetype` column value is `Transfer` (capital T) but topic becomes `banking.Transfer.events` instead of `banking.transfer.events`, creating a new unexpected topic.
**Why it happens:** `transforms.outbox.route.by.field` uses the column value verbatim in the topic name.
**How to avoid:** Ensure all `aggregatetype` values in the outbox_event table are lowercase (e.g., `transfer`, `account`). Add a CHECK constraint in the migration: `CHECK (aggregatetype = lower(aggregatetype))`.
**Warning signs:** Two topics exist: `banking.transfer.events` and `banking.Transfer.events`; one has zero messages.

### Pitfall 5: Spring Kafka Consumer Never Receives Messages (offset reset)

**What goes wrong:** Ledger-service starts after the Kafka topic already has messages. Consumer group `ledger-service` starts with no committed offset. `auto.offset.reset=latest` means all historical messages are skipped.
**Why it happens:** Spring Boot auto-configure defaults `auto-offset-reset` to `latest` in many versions.
**How to avoid:** Set `spring.kafka.consumer.auto-offset-reset=earliest` for all services in Phase 1.1. This ensures messages published before the consumer starts are still processed.
**Warning signs:** Kafka consumer lag is 0 but ledger_entries table is empty after a transfer.

### Pitfall 6: kind Cluster DNS Fails with Rootless Podman

**What goes wrong:** `kubectl exec dns-test -- nslookup kubernetes.default` times out or returns `SERVFAIL`.
**Why it happens:** Rootless Podman on Windows/WSL2 uses user-mode networking (pasta/slirp4netns) which does not support the iptables rules CoreDNS needs.
**How to avoid:** Run the spike with rootful Podman. Current machine state is confirmed rootful: `podman machine inspect` returns `"Rootful": true`. [VERIFIED: environment probe]
**Warning signs:** kind cluster creates successfully but `kubectl get pods -n kube-system` shows `coredns` pods in CrashLoopBackOff.

### Pitfall 7: @RetryableTopic Requires @EnableKafka on Configuration Class

**What goes wrong:** `@RetryableTopic` annotations are ignored; messages go straight to DLT on first failure instead of retrying.
**Why it happens:** Spring Kafka's non-blocking retry requires `@EnableKafka` on a `@Configuration` class (or relies on Spring Boot's auto-configuration).
**How to avoid:** Verify `spring-kafka` is on the classpath — Spring Boot auto-configuration adds `@EnableKafka` automatically when `spring-kafka` is detected. No manual annotation needed if using Spring Boot auto-config. If you configure `KafkaListenerContainerFactory` manually, ensure the auto-configuration is not displaced.
**Warning signs:** Test shows first failure goes directly to `-dlt` topic with zero retries.
