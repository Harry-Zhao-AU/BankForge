### WR-01: Unsafe root.get(field).asText() — NullPointerException on malformed events routes to DLT

**Files modified:**
- `ledger-service/src/main/java/au/com/bankforge/ledger/kafka/LedgerEventListener.java`
- `notification-service/src/main/java/au/com/bankforge/notification/kafka/NotificationEventListener.java`
- `notification-service/src/main/java/au/com/bankforge/notification/kafka/AustracAuditListener.java`

**Commit:** e244292
**Applied fix:** Added explicit null checks before calling `.asText()` on all `root.get("fieldName")` results. All three required fields (`transferId`, `fromAccountId`, `toAccountId`, `amount`) now assign to a local `JsonNode` variable first, then a combined null guard throws `IllegalArgumentException` with the raw payload for diagnostics. In `AustracAuditListener`, the `amount` field is checked before the threshold comparison, and the identity fields are checked inside the `>= THRESHOLD` branch (where they are actually accessed).

### WR-04: pom.xml (ledger-service, notification-service) — DOCKER_HOST hardcoded to Docker named pipe

**File:** `ledger-service/pom.xml:137`, `notification-service/pom.xml:120`
**Reason:** Skipped — the current value `npipe:////./pipe/docker_engine` is the established working pattern for this project. All other pom.xml files in the repository (`account-service/pom.xml`) use the identical value with an explicit comment stating "the machine exposes a named pipe at `npipe:////./pipe/docker_engine`". On Windows, Podman Desktop exposes a Docker-compatible named pipe at this path. Changing to a different path (e.g., `podman-machine-default`) without confirming the actual Podman socket name would break all integration tests. No change applied.
**Original issue:** `DOCKER_HOST=npipe:////./pipe/docker_engine` is a Docker named pipe; project uses Podman.
