# Phase 03 Plan 04: Kong Gateway + Keycloak JWT Auth

**Goal:** All 4 banking services sit behind Kong Gateway. Keycloak issues RS256 JWT tokens. Kong validates every inbound request, strips any forged `X-User-Id` header, and injects a verified one from the JWT `sub` claim. Services receive authenticated requests with no auth code of their own.

---

## Inputs (from Plan 03-03)

- kind cluster `bankforge` running, all pods 2/2 READY
- Istio STRICT mTLS active in `bankforge` namespace
- Observability stack running (Jaeger, Prometheus, Loki, Grafana, Kiali)
- `port-forward.sh` working for UI access

---

## Architecture

```
curl / Postman / MCP
        │
        ▼
  Kong Gateway (kong namespace, no Istio sidecar, DB-less mode)
        │  validates RS256 JWT via Keycloak public key
        │  strips incoming X-User-Id
        │  injects X-User-Id = JWT sub claim
        │
        ▼ (plaintext out of Kong → PERMISSIVE in kong namespace)
  Envoy sidecar (bankforge namespace, mTLS STRICT)
        │
        ▼
  payment-service / account-service / ledger-service / notification-service
        │  reads X-User-Id header — trusted, no JWT parsing needed
```

## Why Helm for Kong only

Kafka, Postgres, Jaeger are stateful infrastructure — a plain Deployment + ConfigMap is sufficient.
Kong is a K8s-aware gateway. The Kong Ingress Controller (KIC) watches `Ingress` and `KongPlugin`
resources and programs Kong's routing table automatically. Without KIC, routes must be registered
manually via the Admin API on every deploy — not reproducible, not GitOps-friendly.

Helm is used once at install time. Everything after that (routes, plugins, JWT config) is plain
YAML committed to the repo, consistent with all other infra.

## Tasks

### Task 1 — Deploy Keycloak

Create `k8s/infrastructure/keycloak.yaml` with four resources:

**Secret** (`keycloak-secret` in `bankforge` namespace):
- `admin-password: admin` (dev only — plain value acceptable for local sandbox)

**ConfigMap** (`keycloak-realm-config`) containing `bankforge-realm.json`:
- Realm: `bankforge`
- Client `kong` — confidential, RS256, used by Kong JWT plugin to request tokens
- Client `bankforge-app` — public, placeholder for future frontend (Authorization Code flow)
- User `testuser` / `password` — role `ROLE_CUSTOMER`
- User `adminuser` / `password` — role `ROLE_ADMIN`
- Token lifetime: 300s, refresh token: 1800s

**Deployment** in `bankforge` namespace:
```yaml
image: quay.io/keycloak/keycloak:26.6.1
args: ["start-dev", "--import-realm"]
env:
  KC_BOOTSTRAP_ADMIN_USERNAME: admin
  KC_BOOTSTRAP_ADMIN_PASSWORD: (from keycloak-secret)
  KC_PROXY_HEADERS: xforwarded
  KC_HTTP_ENABLED: "true"
  KC_HOSTNAME_STRICT: "false"
volumeMounts:
  - name: realm-config
    mountPath: /opt/keycloak/data/import
```

Resource limits (from Research §6):
```
requests: cpu: "500m",  memory: "768Mi"
limits:   cpu: "2000m", memory: "1Gi"
```

Readiness probe: `GET /realms/bankforge`, initialDelaySeconds: 30, periodSeconds: 10, failureThreshold: 6.

**Service** — ClusterIP on port 8080, named `keycloak`.

### Task 2 — Install Kong via Helm

Kong runs in DB-less mode automatically when using KIC — no database required.

```bash
helm repo add kong https://charts.konghq.com
helm repo update

helm install kong kong/ingress \
  -n kong --create-namespace \
  --set controller.ingressController.enabled=true \
  --set podAnnotations."sidecar\.istio\.io/inject"=false
```

Verify:
```bash
kubectl get pods -n kong
kubectl get svc -n kong   # kong-kong-proxy (port 80), kong-kong-admin (port 8001)
```

### Task 4 — Kong plugins + ingress manifests

Create `k8s/kong/`:

**kong-plugins.yaml** — two KongPlugin resources in `bankforge` namespace:

1. `jwt-auth` — validates RS256 JWT on every request:
```yaml
plugin: jwt
config:
  key_claim_name: iss
  claims_to_verify: [exp]
```

2. `inject-user-id` — strips forged `X-User-Id`, injects verified value from JWT `sub`:
```yaml
plugin: request-transformer
config:
  remove:
    headers: [X-User-Id]
  add:
    headers: ["X-User-Id:$(jwt.sub)"]
```

**ingress.yaml** — four Ingress resources in `bankforge` namespace:

| Path prefix | Backend service | Port |
|---|---|---|
| `/api/accounts` | account-service | 8080 |
| `/api/payments` | payment-service | 8080 |
| `/api/ledger` | ledger-service | 8080 |
| `/api/notifications` | notification-service | 8080 |

Each annotated with:
```yaml
annotations:
  konghq.com/plugins: jwt-auth,inject-user-id
  konghq.com/strip-path: "true"
  kubernetes.io/ingress.class: kong
```

### Task 5 — Register Keycloak RS256 public key in Kong

Create `scripts/kong-register-keycloak-key.sh`:

1. Fetch RS256 public key from Keycloak JWKS endpoint (`/realms/bankforge/protocol/openid-connect/certs`)
2. Convert JWK → PEM format (small Python snippet inline in script)
3. Create Kong consumer representing the Keycloak issuer
4. Register JWT credential: algorithm=RS256, key=issuer URL, rsa_public_key=PEM

Script prereqs: port-forwards running for Keycloak (8090) and Kong Admin API (8001).

### Task 8 — Human verify

```bash
# Step 1: Deploy Keycloak
kubectl apply -f k8s/infrastructure/keycloak.yaml
kubectl get pods -n bankforge -l app=keycloak -w
# Wait for Running/Ready

# Step 2: Apply kong namespace PeerAuthentication
kubectl apply -f k8s/istio/kong-permissive.yaml

# Step 3: Install Kong via Helm
helm repo add kong https://charts.konghq.com && helm repo update
helm install kong kong/ingress -n kong --create-namespace \
  --set controller.ingressController.enabled=true \
  --set podAnnotations."sidecar\.istio\.io/inject"=false
kubectl get pods -n kong -w

# Step 4: Apply Kong plugins + ingress routes
kubectl apply -f k8s/kong/

# Step 5: Start port-forwards
bash scripts/port-forward.sh &

# Step 6: Register Keycloak RS256 public key in Kong
bash scripts/kong-register-keycloak-key.sh

# Step 7: Get a JWT token from Keycloak
TOKEN=$(curl -s -X POST \
  http://localhost:8090/realms/bankforge/protocol/openid-connect/token \
  -d "grant_type=password" \
  -d "client_id=kong" \
  -d "client_secret=<kong-client-secret>" \
  -d "username=testuser" \
  -d "password=password" | jq -r '.access_token')

# Step 8: Authenticated call — expect 200
curl -H "Authorization: Bearer $TOKEN" http://localhost:8000/api/accounts/health

# Step 9: No token — expect 401
curl http://localhost:8000/api/accounts/health

# Step 10: Forge X-User-Id — Kong must strip it
curl -H "Authorization: Bearer $TOKEN" \
     -H "X-User-Id: attacker-id" \
     http://localhost:8000/api/accounts/health
# Check service logs: X-User-Id must equal JWT sub, not "attacker-id"

# Step 11: Full validation
bash scripts/validate-auth-k8s.sh
```
