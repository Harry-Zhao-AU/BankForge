# Phase 03 Plan 02: Banking Service Manifests + Istio STRICT mTLS

**Goal:** All 4 Spring Boot banking services running as K8s Deployments with Istio sidecars injected (2/2 READY), connected to their dedicated DBs, then Istio mTLS switched to STRICT — enforcing encrypted, authenticated pod-to-pod traffic.

---

## Inputs (from Plan 03-01)

- kind cluster `bankforge` running
- `bankforge` namespace with `istio-injection: enabled`
- Istio 1.29.2 in PERMISSIVE mode
- 7 infra pods all 2/2 READY: account-db, payment-db, ledger-db, notification-db, redis, kafka, debezium
- DB secrets created: `account-db-secret`, `payment-db-secret`, `ledger-db-secret`, `notification-db-secret`
- Images available at `ghcr.io/harry-zhao-au/bankforge-*:latest`

---

## Tasks

### Task 1 — Write service manifests (4 × YAML)

Create `k8s/services/{account,payment,ledger,notification}-service.yaml`, each containing:
- **ConfigMap** — env vars that override Spring Boot defaults (DB URL, Kafka, Redis, OTel endpoint)
- **Deployment** — 1 replica, envFrom ConfigMap, individual secret env for DB password, resource limits, readinessProbe on /actuator/health
- **Service** — ClusterIP on port 8080

Key env var overrides per service:

| Service | Key overrides |
|---|---|
| account-service | `SPRING_DATASOURCE_URL=jdbc:postgresql://account-db:5432/accountdb`, `SPRING_DATASOURCE_USERNAME=account`, `OTEL_EXPORTER_OTLP_ENDPOINT=http://otel-collector:4318` |
| payment-service | above pattern for payment-db, + `SPRING_KAFKA_BOOTSTRAP_SERVERS=kafka:9092`, `SPRING_DATA_REDIS_HOST=redis`, `SERVICES_ACCOUNT_BASE_URL=http://account-service:8080` |
| ledger-service | ledger-db pattern + `SPRING_KAFKA_BOOTSTRAP_SERVERS=kafka:9092` |
| notification-service | Kafka only (no DB in app config) + OTel |

Resource limits for all services:
```
requests: cpu: "250m", memory: "512Mi"
limits:   cpu: "1000m", memory: "1Gi"
```

Readiness probe: `GET /actuator/health`, initialDelaySeconds: 45, periodSeconds: 10, failureThreshold: 6.

### Task 2 — Write PeerAuthentication STRICT manifest

Create `k8s/istio/peer-authentication-strict.yaml`:
- `PeerAuthentication` in `bankforge` namespace, `mtls.mode: STRICT`
- Applied AFTER all services are 2/2 READY (not before)

### Task 3 — Write port-forward.sh

Create `scripts/port-forward.sh`:
- Forwards all ports needed by the MCP server and local debugging
- Runs all kubectl port-forward commands as background processes
- Trap SIGINT to clean up all PIDs on Ctrl+C

Port mapping:
| Local | K8s Service | Use |
|---|---|---|
| 8081 | account-service:8080 | REST API dev access |
| 8082 | payment-service:8080 | REST API dev access |
| 8083 | ledger-service:8080 | REST API dev access |
| 8084 | notification-service:8080 | REST API dev access |
| 4318 | otel-collector:4318 | OTel HTTP (pre-Plan 03-04) |
| 16686 | jaeger:16686 | Jaeger UI (pre-Plan 03-04) |
| 9090 | prometheus:9090 | Prometheus UI |
| 3000 | grafana:3000 | Grafana UI |
| 20001 | kiali:20001 | Kiali UI |

### Task 4 — Human verify

After human runs apply + STRICT switch:

```bash
# Step 1: Apply service manifests
kubectl apply -f k8s/services/

# Step 2: Wait for 2/2 READY (all 4 services)
kubectl get pods -n bankforge -w

# Step 3: Apply STRICT mTLS
kubectl apply -f k8s/istio/peer-authentication-strict.yaml

# Step 4: Verify services are still 2/2 READY after STRICT
kubectl get pods -n bankforge

# Step 5: Verify mTLS enforcement (expect connection refused/reset)
kubectl run test-curl --image=curlimages/curl --restart=Never -n bankforge \
  --command -- curl -sf http://account-service:8080/actuator/health
# Expected: fails (no Istio cert) OR add --rm for cleanup

# Step 6: Verify legit Istio pod can reach account-service
kubectl exec -n bankforge deploy/payment-service -- \
  curl -sf http://account-service:8080/actuator/health

# Step 7: Confirm PeerAuthentication is STRICT
kubectl get peerauthentication -n bankforge
```
