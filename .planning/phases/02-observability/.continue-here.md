# BLOCKING CONSTRAINTS — Read Before Anything Else

> These are not suggestions. Each constraint below was discovered through failure.
> Acknowledge each one explicitly before proceeding.

- [ ] CONSTRAINT: Wrong OTel logging property — `management.otlp.logging.export.endpoint` does NOT exist in Spring Boot 4.0.5. The correct key (confirmed from JAR metadata) is `management.opentelemetry.logging.export.otlp.endpoint`. Commit 6ba6aed introduced this regression. All 4 application.yml files must be fixed before rebuilding.
- [ ] CONSTRAINT: OTel SDK does NOT use standard `OTEL_*` env vars — Spring Boot 4.x builds the SDK from Spring customizer beans only. All export configuration must be via `management.opentelemetry.*` properties, not env vars alone.

**Do not proceed until all boxes are checked.**

## Critical Anti-Patterns

| Pattern | Description | Severity | Prevention Mechanism |
|---------|-------------|----------|---------------------|
| Wrong logging property key (2nd occurrence) | commit 6ba6aed changed `management.opentelemetry.logging.export.otlp.endpoint` → `management.otlp.logging.export.endpoint`. The second form does not exist in SB 4.0.5. This was a regression introduced while trying to fix the first occurrence. | blocking | Always verify Spring Boot 4.x property keys directly from JAR metadata (`META-INF/spring-configuration-metadata.json`) before committing any management.* property change. |
| Smoke test `((PASS++))` under `set -e` | `((0++))` evaluates to 0 (falsy), killing the script after the first check when PASS starts at 0. Fixed: use `PASS=$((PASS + 1))`. | advisory | Use `VAR=$((VAR + 1))` pattern in bash scripts with `set -e`, never `((VAR++))` on a counter that starts at 0. |
| OTel Collector health check with `curl -sf` on OTLP receiver | Port 4318 returns 405 on GET — correct behaviour for an OTLP receiver, but `-sf` treats 4xx as failure. Fixed: check HTTP response code is not 000 (connection refused). | advisory | For non-API service health checks, check connection success (not 2xx) using `curl -s -o /dev/null -w '%{http_code}'` and compare to `000`. |
| Distroless image healthchecks (3rd recurrence) | Loki 3.7.0 image has no shell, no nc — CMD-SHELL healthchecks silently fail. Fixed: remove healthcheck (no service depends on `service_healthy` for Loki). | advisory | Before writing a healthcheck for any Grafana image, exec into it and check for shell/tools. If distroless, remove or disable healthcheck. |
