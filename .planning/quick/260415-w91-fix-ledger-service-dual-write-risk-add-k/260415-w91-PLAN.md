<objective>
Fix the ledger-service dual-write risk where a crash between DB commit and Kafka send causes duplicate ledger entries on retry.

Purpose: Eliminate financial data corruption risk — duplicate debit/credit entries violate double-entry bookkeeping invariants. The fix adds both a database-level idempotency guard (UNIQUE constraint + skip-if-exists) and Kafka Exactly-Once Semantics (EOS) so consumer offset commit and producer send are atomically bound.

Output: Idempotent ledger writes, Kafka EOS on the ledger listener, updated integration test proving duplicate delivery is safe.

**B. Create `KafkaConfig.java`** in `au.com.bankforge.ledger.config`:

```java
package au.com.bankforge.ledger.config;

import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.kafka.config.ConcurrentKafkaListenerContainerFactory;
import org.springframework.kafka.core.ConsumerFactory;
import org.springframework.kafka.core.KafkaTemplate;
import org.springframework.kafka.core.ProducerFactory;
import org.springframework.kafka.transaction.KafkaTransactionManager;
import org.springframework.kafka.listener.ContainerProperties;

@Configuration(proxyBeanMethods = false)
public class KafkaConfig {

    @Bean
    public KafkaTransactionManager<String, String> kafkaTransactionManager(
            ProducerFactory<String, String> producerFactory) {
        return new KafkaTransactionManager<>(producerFactory);
    }

    @Bean
    public ConcurrentKafkaListenerContainerFactory<String, String> kafkaListenerContainerFactory(
            ConsumerFactory<String, String> consumerFactory,
            KafkaTransactionManager<String, String> kafkaTransactionManager,
            KafkaTemplate<String, String> kafkaTemplate) {
        ConcurrentKafkaListenerContainerFactory<String, String> factory =
                new ConcurrentKafkaListenerContainerFactory<>();
        factory.setConsumerFactory(consumerFactory);
        factory.getContainerProperties().setTransactionManager(kafkaTransactionManager);
        factory.getContainerProperties().setEosMode(ContainerProperties.EOSMode.V2);
        factory.setReplyTemplate(kafkaTemplate);
        return factory;
    }
}
```

This wires the Kafka transaction manager into the listener container so that consumer offset commit + producer send are in the same Kafka transaction. EOSMode.V2 uses the `group.instance.id` fencing approach (Kafka 2.5+), which is the current default and avoids the deprecated V1 per-consumer-instance producer overhead.

IMPORTANT: Do NOT set `factory.setReplyTemplate()` unless needed. Actually, remove `setReplyTemplate` — the template is used directly in the listener, not as a reply template. The `kafkaTemplate` is already transactional because `transaction-id-prefix` is set.

**C. Update `LedgerEventListener.onTransferEvent()`** — add idempotency guard at the top of the method, BEFORE the save calls:

After parsing `transferId` from the payload and before building the debit/credit entries, add:

```java
if (ledgerEntryRepository.existsByTransferId(transferId)) {
    log.info("Ledger entries already exist for transferId={}, skipping (idempotent replay)", transferId);
    // Still publish confirmation — the downstream consumer may not have received it
    String confirmPayload = "{\"transferId\":\"" + transferId + "\",\"status\":\"CONFIRMED\"}";
    kafkaTemplate.send("banking.transfer.confirmed", transferId.toString(), confirmPayload);
    return;
}
```

This means: on replay, we skip DB writes (they already exist) but still send the confirmation event (which is safe because the Kafka transaction is idempotent). The `@Transactional` annotation should REMAIN on the method — it still provides the DB transaction for the non-replay path. The Kafka transaction is managed by the container factory's `KafkaTransactionManager`, wrapping the entire listener invocation.

Also remove `@RetryableTopic` from the listener method. `@RetryableTopic` creates retry topics and DLT topics using non-transactional internal producers that CONFLICT with EOS. Instead, configure retry via the container factory's error handler. For now, the idempotency guard + EOS makes simple retry safe. The `@DltHandler` method should also be removed (it was paired with `@RetryableTopic`).

Replace `@RetryableTopic` + `@DltHandler` with a `DefaultErrorHandler` on the container factory in `KafkaConfig.java`. Add to the factory configuration in KafkaConfig:

```java
import org.springframework.kafka.listener.DefaultErrorHandler;
import org.springframework.util.backoff.ExponentialBackOff;

// Inside kafkaListenerContainerFactory method, before return:
ExponentialBackOff backOff = new ExponentialBackOff(1000L, 2.0);
backOff.setMaxElapsedTime(30000L);
factory.setCommonErrorHandler(new DefaultErrorHandler(backOff));
```

This preserves the retry-with-backoff behavior without conflicting with EOS.
