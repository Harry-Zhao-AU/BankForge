## Objective

Add transfer_initiated_total metric counters for the CONFIRMED and FAILED states so the full
transfer state funnel (PENDING, POSTING, CONFIRMED, CANCELLED, FAILED) is observable in
Grafana without querying the database.

**Purpose:** Currently CONFIRMED (from TransferConfirmationListener) and FAILED (from HungTransferDetector)
are invisible to Prometheus. This gap means Grafana dashboards cannot show the complete transfer
lifecycle or alert on confirmation rate drops.

**Output:** Two modified production files with MeterRegistry injected and counter increments, plus
updated unit test for HungTransferDetector.

### Interfaces (metric pattern to replicate)

The established pattern from PaymentService.java:

```java
private void incrementTransferInitiated(TransferState state) {
    Counter.builder("transfer_initiated_total")
        .description("Number of transfers initiated")
        .tag("service", "payment-service")
        .tag("state", state.name())
        .register(meterRegistry)
        .increment();
}
```

Both TransferConfirmationListener and HungTransferDetector use @RequiredArgsConstructor (Lombok),
so adding a `private final MeterRegistry meterRegistry` field auto-wires via constructor injection.

### Task 2: Update HungTransferDetectorTest to inject MeterRegistry and verify metric increments

**Type:** auto

**Files:**
- payment-service/src/test/java/au/com/bankforge/payment/detector/HungTransferDetectorTest.java

**Action:**

The existing test constructs HungTransferDetector manually with 3 args:
`new HungTransferDetector(transferRepository, transferStateService, kafkaTemplate)`

Adding `MeterRegistry` as a 4th constructor field (via Lombok) breaks this.

1. Add imports: `import io.micrometer.core.instrument.MeterRegistry;` and `import io.micrometer.core.instrument.simple.SimpleMeterRegistry;`
2. Add field: `private MeterRegistry meterRegistry;` (NOT @Mock -- use real SimpleMeterRegistry so counters actually work)
3. In setUp(), add `meterRegistry = new SimpleMeterRegistry();` before the constructor call
4. Update the constructor call to include the 4th arg:
   `new HungTransferDetector(transferRepository, transferStateService, kafkaTemplate, meterRegistry);`
5. In `shouldCancelHungPaymentProcessingTransfers()`, add assertion after existing verifies:
   ```java
   assertThat(meterRegistry.get("transfer_initiated_total")
       .tag("state", "CANCELLED").counter().count()).isEqualTo(1.0);
   ```
6. In `shouldMarkHungPostingAsFailed()`, add assertion after existing verifies:
   ```java
   assertThat(meterRegistry.get("transfer_initiated_total")
       .tag("state", "FAILED").counter().count()).isEqualTo(1.0);
   ```
7. In `shouldDoNothingWhenNoHungTransfers()`, add assertion that no transfer_initiated_total counters exist:
   ```java
   assertThat(meterRegistry.find("transfer_initiated_total").counters()).isEmpty();
   ```

NOTE: The constructor arg order depends on the field declaration order in HungTransferDetector.java.
Since @RequiredArgsConstructor generates the constructor in field declaration order, and we are
adding `private final MeterRegistry meterRegistry;` as the LAST field, meterRegistry will be the
4th constructor argument (after transferRepository, transferStateService, kafkaTemplate).
